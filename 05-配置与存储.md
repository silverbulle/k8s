# 05 - 配置与存储

应用运行需要配置（环境变量、配置文件）和数据持久化（数据库、文件存储）。K8s 将配置和存储从容器镜像中解耦，通过独立的资源对象管理。

---

## 5.1 ConfigMap

ConfigMap 以键值对形式存储非敏感配置数据，注入 Pod 供应用使用。

### 创建方式

```bash
# 从文件创建
kubectl create configmap app-config \
  --from-file=config.properties \
  --from-file=nginx.conf

# 从字面值创建
kubectl create configmap app-config \
  --from-literal=DATABASE_URL=mysql://localhost:3306/app \
  --from-literal=LOG_LEVEL=info

# 从 YAML 创建
kubectl apply -f - <<EOF
apiVersion: v1
kind: ConfigMap
metadata:
  name: app-config
data:
  DATABASE_URL: "mysql://localhost:3306/app"
  LOG_LEVEL: "info"
  nginx.conf: |
    server {
      listen 80;
      location / { proxy_pass http://backend:8080; }
    }
EOF
```

### 注入 Pod

```yaml
spec:
  containers:
  - name: app
    image: myapp:1.0

    # 方式一：注入为环境变量
    env:
    - name: DATABASE_URL
      valueFrom:
        configMapKeyRef:
          name: app-config
          key: DATABASE_URL

    # 方式二：批量注入所有键值对为环境变量
    envFrom:
    - configMapRef:
        name: app-config

    # 方式三：挂载为文件
    volumeMounts:
    - name: config-volume
      mountPath: /etc/config
      readOnly: true

  volumes:
  - name: config-volume
    configMap:
      name: app-config
      items:                    # 可选：只挂载特定 key
      - key: nginx.conf
        path: nginx/nginx.conf  # 挂载到 /etc/config/nginx/nginx.conf
```

### 热更新

ConfigMap 以 Volume 挂载时，修改 ConfigMap 后文件会自动更新（延迟约 1-2 分钟）。但以环境变量注入时**不会自动更新**，需要重启 Pod。

```bash
# 更新 ConfigMap
kubectl edit configmap app-config

# 触发 Deployment 滚动更新（使环境变量生效）
kubectl rollout restart deployment/app
```

## 5.2 Secret

Secret 存储敏感数据（密码、Token、证书），以 Base64 编码存储（并非加密）。

### 类型

| 类型 | 用途 |
|------|------|
| `Opaque`（默认） | 通用敏感数据 |
| `kubernetes.io/tls` | TLS 证书和密钥 |
| `kubernetes.io/dockerconfigjson` | 镜像仓库认证 |
| `kubernetes.io/basic-auth` | Basic Auth |
| `kubernetes.io/ssh-auth` | SSH 密钥 |
| `kubernetes.io/service-account-token` | ServiceAccount Token |

### 创建方式

```bash
# 通用 Secret
kubectl create secret generic db-secret \
  --from-literal=username=admin \
  --from-literal=password='S3cur3P@ss'

# TLS Secret
kubectl create secret tls tls-secret \
  --cert=tls.crt --key=tls.key

# 镜像仓库认证
kubectl create secret docker-registry registry-secret \
  --docker-server=registry.example.com \
  --docker-username=admin \
  --docker-password='password'
```

### 注入 Pod

```yaml
spec:
  containers:
  - name: app
    image: myapp:1.0

    # 环境变量引用
    env:
    - name: DB_PASSWORD
      valueFrom:
        secretKeyRef:
          name: db-secret
          key: password

    # 批量注入
    envFrom:
    - secretRef:
        name: db-secret

    # 挂载为文件
    volumeMounts:
    - name: secret-volume
      mountPath: /etc/secrets
      readOnly: true

  volumes:
  - name: secret-volume
    secret:
      secretName: db-secret

  # 镜像拉取认证
  imagePullSecrets:
  - name: registry-secret
```

### 安全增强

Base64 不是加密。生产环境建议：

1. **启用 etcd 静态加密**（EncryptionConfiguration）
2. **使用外部密钥管理**（Vault、AWS Secrets Manager、External Secrets Operator）
3. **启用 RBAC 严格控制 Secret 访问权限**

## 5.3 Volume

Volume 为 Pod 中的容器提供共享存储。Pod 内所有容器可以挂载同一个 Volume。

### 常用 Volume 类型

| 类型 | 说明 | 生命周期 |
|------|------|---------|
| **emptyDir** | 临时目录，Pod 内共享 | 随 Pod 删除 |
| **hostPath** | 挂载宿主机目录 | 随 Node 存在 |
| **configMap** | 挂载 ConfigMap 为文件 | 随 ConfigMap |
| **secret** | 挂载 Secret 为文件 | 随 Secret |
| **persistentVolumeClaim** | 持久化存储 | 独立于 Pod |
| **nfs** | NFS 共享存储 | 随 NFS 服务器 |

### emptyDir

```yaml
# 容器间共享临时数据
volumes:
- name: shared-data
  emptyDir:
    medium: ""          # "" = 磁盘, "Memory" = tmpfs（内存）
    sizeLimit: 1Gi      # 限制大小
```

典型用途：Sidecar 日志收集、init container 生成配置文件。

### hostPath

```yaml
volumes:
- name: docker-socket
  hostPath:
    path: /var/run/docker.sock
    type: Socket
```

**注意：** hostPath 绑定到特定 Node，Pod 调度到其他 Node 后无法访问原数据。通常仅用于 DaemonSet（如日志收集）。

## 5.4 PersistentVolume (PV) 与 PersistentVolumeClaim (PVC)

PV/PVC 将存储供应与使用解耦：集群管理员提供 PV（存储资源），应用通过 PVC 申请使用。

### 关系模型

```
集群管理员创建 PV（存储资源池）
        │
        ▼
┌─────────────────┐     绑定      ┌──────────────────┐
│  Persistent     │◄────────────►│  Persistent      │
│  Volume (PV)    │              │  VolumeClaim     │
│                 │              │  (PVC)            │
│ 100Gi, SSD      │              │ 申请 50Gi, SSD    │
│ NFS/iSCSI/云盘  │              │                  │
└─────────────────┘              └──────────────────┘
                                         │
                                         ▼
                                  Pod 挂载使用
```

### PV 示例

```yaml
apiVersion: v1
kind: PersistentVolume
metadata:
  name: pv-nfs-001
spec:
  capacity:
    storage: 100Gi
  accessModes:
  - ReadWriteMany            # 多节点读写
  persistentVolumeReclaimPolicy: Retain   # 删除 PVC 后保留数据
  nfs:
    server: 192.168.1.100
    path: /exports/data
---
# 云盘 PV（以 AWS EBS 为例）
apiVersion: v1
kind: PersistentVolume
metadata:
  name: pv-ebs-001
spec:
  capacity:
    storage: 50Gi
  accessModes:
  - ReadWriteOnce             # 单节点读写
  awsElasticBlockStore:
    volumeID: vol-0abc123
    fsType: ext4
```

### 访问模式

| 模式 | 缩写 | 说明 |
|------|------|------|
| ReadWriteOnce | RWO | 单节点读写 |
| ReadOnlyMany | ROX | 多节点只读 |
| ReadWriteMany | RWX | 多节点读写 |
| ReadWriteOncePod | RWOP | 单 Pod 读写（1.22+） |

### PVC 示例

```yaml
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: my-pvc
spec:
  accessModes:
  - ReadWriteOnce
  resources:
    requests:
      storage: 50Gi
  storageClassName: standard        # 关联 StorageClass
```

### PV 回收策略

| 策略 | 行为 |
|------|------|
| **Retain** | 删除 PVC 后 PV 保留，数据需手动清理 |
| **Delete**（默认） | 删除 PVC 时同时删除 PV 和底层存储 |
| **Recycle**（已废弃） | 执行 `rm -rf /volume/*` 后回收 |

### 在 Pod 中使用 PVC

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: app-with-storage
spec:
  containers:
  - name: app
    image: myapp:1.0
    volumeMounts:
    - name: data
      mountPath: /app/data
  volumes:
  - name: data
    persistentVolumeClaim:
      claimName: my-pvc
```

## 5.5 StorageClass

StorageClass 实现了**动态供给（Dynamic Provisioning）**——无需管理员预先创建 PV，PVC 创建时自动创建对应的 PV。

### 工作原理

```
用户创建 PVC (storageClassName: fast)
  │
  ▼
StorageClass "fast" (provisioner: ebs.csi.aws.com)
  │
  ▼
自动创建 PV + 底层存储（AWS EBS gp3 卷）
  │
  ▼
PVC 自动绑定到新 PV
```

### 常见 StorageClass

```yaml
# AWS EBS
apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: ebs-gp3
provisioner: ebs.csi.aws.com
parameters:
  type: gp3
  encrypted: "true"
reclaimPolicy: Delete
allowVolumeExpansion: true     # 支持在线扩容
volumeBindingMode: WaitForFirstConsumer  # 等 Pod 调度后再创建

---
# NFS
apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: nfs
provisioner: nfs.csi.k8s.io
parameters:
  server: 192.168.1.100
  share: /exports
reclaimPolicy: Retain

---
# 本地存储（高性能场景）
apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: local-storage
provisioner: kubernetes.io/no-provisioner
volumeBindingMode: WaitForFirstConsumer
```

### volumeBindingMode

| 模式 | 说明 |
|------|------|
| `Immediate`（默认） | PVC 创建后立即绑定 PV |
| `WaitForFirstConsumer` | 等使用 PVC 的 Pod 被调度后，再根据 Pod 所在 Node 创建 PV（推荐） |

`WaitForFirstConsumer` 解决了存储拓扑约束问题——比如 AWS EBS 卷只能挂载到同 AZ 的 Node，如果先创建 PV 再调度 Pod，可能调度到不同 AZ 导致挂载失败。

### 设置默认 StorageClass

```yaml
apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: standard
  annotations:
    storageclass.kubernetes.io/is-default-class: "true"
```

PVC 不指定 `storageClassName` 时使用默认 StorageClass。

## 存储选择指南

| 场景 | 推荐方案 | 理由 |
|------|---------|------|
| 数据库（MySQL/PostgreSQL） | 云盘 StorageClass（RWO） | 高性能、低延迟 |
| 共享文件（多 Pod 读写） | NFS / EFS StorageClass（RWX） | 支持多节点挂载 |
| 临时缓存 / 编译产物 | emptyDir | 轻量、随 Pod 清理 |
| 日志收集 | hostPath + DaemonSet | 读取节点上的日志文件 |
| 大数据 / AI 训练 | 本地 SSD + Local PV | 最高 I/O 性能 |
| 对象存储（图片/视频） | 不走 PV，直接用 SDK 访问 S3/OSS | 避免文件系统抽象开销 |

## 本章小结

| 资源 | 一句话总结 | 关键概念 |
|------|-----------|---------|
| ConfigMap | 非敏感配置注入 | 环境变量/文件挂载, 热更新 |
| Secret | 敏感数据管理 | Base64, TLS/Docker, 外部密钥管理 |
| Volume | Pod 内容器共享存储 | emptyDir, hostPath |
| PV/PVC | 持久化存储申请与绑定 | 访问模式, 回收策略 |
| StorageClass | 动态存储供给 | CSI Provisioner, WaitForFirstConsumer |
