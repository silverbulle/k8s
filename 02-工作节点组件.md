# 02 - 工作节点组件

工作节点（Worker Node）是 K8s 集群中真正运行工作负载的机器。每个 Worker 节点运行三个核心组件：kubelet、kube-proxy 和容器运行时。

```
┌────────────────────────────────────────────────────┐
│                    Worker Node                      │
│                                                    │
│  ┌──────────┐   ┌──────────┐   ┌───────────────┐  │
│  │ kubelet  │   │kube-proxy│   │ Container RT  │  │
│  │          │   │          │   │ (containerd)  │  │
│  │ Pod 生命 │   │ 网络规则 │   │               │  │
│  │ 周期管理 │   │ 管理     │   │ 运行容器      │  │
│  └──────────┘   └──────────┘   └───────────────┘  │
│                                                    │
│  ┌─────────┐ ┌─────────┐ ┌─────────┐             │
│  │  Pod A  │ │  Pod B  │ │  Pod C  │             │
│  └─────────┘ └─────────┘ └─────────┘             │
└────────────────────────────────────────────────────┘
```

## 2.1 kubelet

kubelet 是每个 Worker 节点上的"节点代理"，负责 Pod 的生命周期管理。它从 apiserver 接收 PodSpec，确保容器按照期望状态运行。

### 核心职责

| 职责 | 说明 |
|------|------|
| **Pod 生命周期** | 创建/启动/停止/删除容器 |
| **健康检查** | 执行 Liveness/Readiness/Startup Probe |
| **状态上报** | 定期向 apiserver 汇报 Pod 和 Node 状态 |
| **Volume 管理** | 挂载/卸载 Pod 所需的 Volume |
| **镜像拉取** | 通过容器运行时拉取镜像 |
| **日志与 exec** | 提供容器日志查询和 exec 入口 |

### kubelet 如何获取 Pod 配置

kubelet 通过三种方式获取 Pod 配置（PodSpec）：

1. **API Server（主要方式）**：Watch `/api/v1/pods`，获取调度到本节点的 Pod
2. **Static Pod**：读取本地 manifest 目录（默认 `/etc/kubernetes/manifests/`），kube-apiserver 等控制平面组件就是这种方式运行的
3. **HTTP URL（已废弃）**：从 URL 定期拉取 Pod 配置

### Static Pod — 静态 Pod

上面第 2 条提到的 Static Pod 值得展开。**它不是一种"工作负载资源"**（Deployment/StatefulSet 那套有控制器、有 replicas/selector 的体系），而是一种 **Pod 的创建来源**——由 **kubelet 直接从节点本地 manifest 文件创建**，不经 apiserver。

| 维度 | 工作负载资源（Deployment/StatefulSet/...） | Static Pod |
|------|----------------------------------------|-----------|
| 谁创建 Pod | 控制器 watch apiserver 后创建 | **kubelet 直接读本地 manifest 文件创建** |
| 经 apiserver 创建吗 | 经（用户提交 yaml → apiserver → controller） | **不经 apiserver 创建** |
| yaml 存哪 | 集群 etcd 里 | 节点本地目录（默认 `/etc/kubernetes/manifests/`） |
| 谁管生命周期 | 对应 controller | **本节点 kubelet**，挂了自动重新拉起 |
| apiserver 能控制吗 | 能（可 kubectl scale/delete） | **不能**，apiserver 里只是只读镜像 |

#### 只读镜像机制

kubelet 把 static pod 起起来后，会**反向同步**一个 Pod 对象到 apiserver，所以 `kubectl get pods -n kube-system` 能看到它们。但这个对象是**只读镜像**——apiserver 不能真正控制它：

```bash
# 看到 kube-system 里带这个注解的都是 static pod
kubectl get pod etcd-master -n kube-system -o jsonpath='{.metadata.annotations}'
# 里面有 "kubernetes.io/config.source": "file"

# 尝试 kubectl delete 会被 kubelet 立即重新拉起，删不掉
kubectl delete pod etcd-master -n kube-system
# → 删除后几秒内又自动重建
```

**真正删除/修改 static pod 的唯一方式是操作节点上的 manifest 文件**——删文件则 kubelet 自动清理对应 Pod，改文件则 kubelet 重新加载。

#### 典型用途

最重要的用途是**自建集群的控制平面组件**。kubeadm 集群里 `kube-apiserver`、`etcd`、`kube-scheduler`、`kube-controller-manager` 都是 static pod：

```bash
# 查看控制平面 manifest 文件
ls /etc/kubernetes/manifests/
# etcd.yaml  kube-apiserver.yaml  kube-controller-manager.yaml  kube-scheduler.yaml
```

为什么控制平面组件必须用 static pod？因为 **apiserver 自己还没起来时，不能用 apiserver 管理自己**——static pod 只依赖 kubelet 和本地文件，能在 apiserver 不可用时仍把控制平面拉起来，这是 bootstrap 的关键。节点本地的监控 agent、日志 agent 也常用 static pod（避免 agent 挂了没人拉起）。

#### 与 DaemonSet 的区别（易混）

两者都能实现"每个节点跑一个"，但层次不同：

| | DaemonSet | Static Pod |
|--|-----------|-----------|
| 经 apiserver | ✅ | ❌ |
| 有 controller | ✅（daemonset controller） | ❌ |
| 跨所有节点 | ✅ 自动 | 仅 manifest 文件所在节点 |
| kubectl 管理 | ✅ | ❌（要登节点改文件） |
| 依赖 apiserver 存活 | ✅ | ❌ |

**选择原则**：需要 apiserver 集中管理、滚动更新 → DaemonSet；需要不依赖 apiserver、节点自举（如控制平面自身）→ Static Pod。

### 探针机制（Probes）

kubelet 通过探针监控容器的健康状态：

```yaml
containers:
- name: app
  image: myapp:1.0
  
  # Startup Probe - 容器启动期间检测（成功后其他探针才生效）
  startupProbe:
    httpGet:
      path: /healthz
      port: 8080
    failureThreshold: 30       # 最多允许 30 次失败
    periodSeconds: 10          # 每 10 秒检查一次 = 最多等 5 分钟启动

  # Liveness Probe - 判断容器是否需要重启
  livenessProbe:
    httpGet:
      path: /healthz
      port: 8080
    initialDelaySeconds: 5
    periodSeconds: 10
    failureThreshold: 3        # 连续 3 次失败 → 重启容器

  # Readiness Probe - 判断容器是否可以接收流量
  readinessProbe:
    httpGet:
      path: /ready
      port: 8080
    periodSeconds: 5
    failureThreshold: 3        # 连续 3 次失败 → 从 Service Endpoints 移除
```

**三种探针对比：**

| 探针类型 | 失败动作 | 典型用途 |
|---------|---------|---------|
| Liveness | 重启容器 | 检测死锁、无限循环 |
| Readiness | 从 Service 摘除 | 应用预热、依赖就绪检测 |
| Startup | 停止其他探针直到成功 | 慢启动应用（Java 等） |

**探针支持四种检测方式：**

```yaml
# 1. HTTP GET
httpGet:
  path: /healthz
  port: 8080
  httpHeaders:
  - name: Custom-Header
    value: value

# 2. TCP Socket
tcpSocket:
  port: 3306

# 3. Exec 命令
exec:
  command:
  - cat
  - /tmp/healthy

# 4. gRPC（1.24+）
grpc:
  port: 50051
```

### Pod 生命周期阶段

```
Pending → Running → Succeeded
                  → Failed

Pending:   调度中或镜像拉取中
Running:   至少一个容器在运行
Succeeded: 所有容器正常退出（Job 类 Pod）
Failed:    至少一个容器异常退出
Unknown:   无法获取 Pod 状态（通常节点故障）
```

### 节点状态与驱逐

kubelet 监控节点资源压力，触发驱逐（Eviction）：

```bash
# 查看节点状态
kubectl describe node <node-name>

# Conditions 说明
Ready:              True   # 节点健康，可以接受 Pod
MemoryPressure:     False  # 内存不足
DiskPressure:       False  # 磁盘不足
PIDPressure:        False  # 进程数过多
NetworkUnavailable: False  # 网络未配置
```

**驱逐优先级**（当资源压力增大时）：

1. 先驱逐 BestEffort QoS 的 Pod
2. 再驱逐 Burstable QoS 的 Pod
3. 最后驱逐 Guaranteed QoS 的 Pod

### kubelet 配置

```yaml
# kubelet-config.yaml（常用配置项）
apiVersion: kubelet.config.k8s.io/v1beta1
kind: KubeletConfiguration

# 资源预留
kubeReserved:          # K8s 系统组件预留
  cpu: "100m"
  memory: "256Mi"
systemReserved:        # 操作系统预留
  cpu: "100m"
  memory: "256Mi"

# 驱逐阈值
evictionHard:
  memory.available: "100Mi"
  nodefs.available: "10%"
  imagefs.available: "15%"

# 镜像 GC
imageGCHighThresholdPercent: 85
imageGCLowThresholdPercent: 80

# 最大 Pod 数（默认 110）
maxPods: 110
```

### 端口

| 协议 | 端口 | 说明 |
|------|------|------|
| **TCP** | **10250** | kubelet HTTPS 端口。提供 metrics 指标、exec 进入容器（`kubectl exec` 底层调用此端口）、logs 查询等 API。apiserver 通过此端口与 kubelet 通信 |
| **TCP** | **10248** | kubelet 本地健康检查端口（`/healthz`）。默认仅绑定 `localhost`，供本机探针检测 kubelet 是否存活 |

> 10250 是集群内最敏感的端口之一——拥有此端口访问权限等同于可以在任意 Node 上执行命令。务必通过 RBAC 和 mTLS 保护。

## 2.2 kube-proxy

kube-proxy 在每个 Worker 节点上维护网络规则，实现 **Service 抽象**——让 Pod 间通信不依赖具体 IP。

### 工作模式

| 模式 | 实现方式 | 适用场景 |
|------|---------|---------|
| **iptables**（默认） | 每个 Service 对应一组 iptables 规则 | 中小规模集群（< 1000 Service） |
| **IPVS** | Linux IPVS 内核模块，哈希表查找 | 大规模集群（> 1000 Service） |
| **nftables**（1.29+） | nftables 替代 iptables | 新一代 Linux 防火墙 |

### iptables 模式原理

当创建 Service 时，kube-proxy 在本节点写入 iptables 规则：

```
# 访问 Service ClusterIP → DNAT 到后端 Pod IP
-A KUBE-SERVICES -d 10.96.0.10/32 -p tcp --dport 53 -j KUBE-SVC-XXXXX
-A KUBE-SVC-XXXXX -m statistic --mode random --probability 0.33 -j KUBE-SEP-1  # 1/3 → Pod A
-A KUBE-SVC-XXXXX -m statistic --mode random --probability 0.50 -j KUBE-SEP-2  # 1/2 → Pod B
-A KUBE-SVC-XXXXX -j KUBE-SEP-3                                                 # 1/1 → Pod C
```

**关键理解：** Service 的 ClusterIP 是一个虚拟 IP，不绑定任何网络接口。流量通过 iptables DNAT 直接转发到后端 Pod，不经过中间代理。

### IPVS 模式原理

```bash
# IPVS 虚拟服务
ipvsadm -ln
IP Virtual Server version 1.2.1
Prot LocalAddress:Port Scheduler Flags
  -> RemoteAddress:Port           Forward Weight ActiveConn InActConn
TCP  10.96.0.10:53 rr             # ClusterIP:53 → round-robin
  -> 10.244.1.5:53                Masq    1      0          0
  -> 10.244.2.8:53                Masq    1      0          0
  -> 10.244.3.2:53                Masq    1      0          0
```

IPVS 支持多种负载均衡算法：rr（轮询）、wrr（加权轮询）、lc（最少连接）、wlc（加权最少连接）、sh（源哈希）、sed（最短期望延迟）、nq（不排队调度）。

### kube-proxy 配置

```yaml
apiVersion: kubeproxy.config.k8s.io/v1alpha1
kind: KubeProxyConfiguration
mode: "ipvs"              # 切换为 IPVS 模式

ipvs:
  scheduler: "rr"         # 负载均衡算法
  minSyncPeriod: "0s"
  syncPeriod: "30s"

# conntrack 配置
conntrack:
  maxPerCore: 32768       # 每个 CPU 核心最大连接跟踪数
  min: 131072
  tcpCloseWaitTimeout: "1h0m0s"
```

### 端口

| 协议 | 端口 | 说明 |
|------|------|------|
| **TCP** | **10249** | kube-proxy metrics 端口（`/metrics`）。默认仅绑定 `localhost`，暴露 Prometheus 格式的指标供本地采集 |
| **TCP** | **10256** | kube-proxy 健康检查端口（`/healthz`）。可供外部负载均衡器或监控系统探测 kube-proxy 自身是否健康 |

> IPVS 模式下，kube-proxy 还会在节点上创建 LVS 规则，并通过 10256 端口探测每个 endpoint 是否健康来决定是否继续向其转发流量。

## 2.3 容器运行时

容器运行时（Container Runtime）是真正负责运行容器的底层组件。K8s 通过 **CRI（Container Runtime Interface）** 与运行时交互。

### CRI 架构

```
kubelet
  │
  │ gRPC (CRI 接口)
  ▼
┌─────────────────────────┐
│   Container Runtime     │
│                         │
│  RuntimeService:        │
│    RunPodSandbox()      │
│    CreateContainer()    │
│    StartContainer()     │
│    StopContainer()      │
│    RemoveContainer()    │
│                         │
│  ImageService:          │
│    PullImage()          │
│    ListImages()         │
│    RemoveImage()        │
└─────────────────────────┘
```

### 主流运行时对比

| 运行时 | 维护方 | 特点 |
|--------|--------|------|
| **containerd** | CNCF | 轻量、稳定，目前主流选择 |
| **CRI-O** | Red Hat | 专为 K8s 设计，最小化攻击面 |
| **Docker Engine** | Docker | 通过 cri-dockerd 适配器支持 CRI |

### containerd 架构

```
containerd
  ├── shim（每个 Pod 一个）
  │   └── runc（创建容器）
  │       └── 容器进程
  ├── shim
  │   └── runc
  │       └── 容器进程
  └── ...
```

**shim 进程的作用：** containerd 重启时，shim 保持容器运行不中断，实现守护进程的无状态化。

### 镜像拉取流程

```
kubelet 收到 PodSpec（含镜像名）
  │
  ├─ 本地有镜像？──是──→ 直接使用
  │
  └─ 没有？──→ imagePullSecrets 认证
               │
               └─→ CRI PullImage() → containerd/docker pull
                    │
                    └─→ 从 Registry 拉取镜像层
```

**imagePullPolicy 策略：**

| 策略 | 行为 |
|------|------|
| `Always` | 每次都拉取（`latest` 标签默认值） |
| `IfNotPresent` | 本地没有才拉取（指定版本标签默认值） |
| `Never` | 只使用本地镜像 |

## 2.4 APIServer 高可用本地代理 (Local Proxy)

> 说明：这是一个**可选的 HA 实践**，并非工作节点的核心组件（核心仍是 kubelet、kube-proxy、容器运行时三件套）。它解决的是"工作节点如何高可用地访问多 Master apiserver"这一问题，与 [4.4 NodeLocal DNSCache](04-服务与网络.md#nodelocal-dnscache--dns-性能优化) 思路相近但有本质区别：**DNSCache 是"本地缓存命中即返回"，而本节方案是"仅负载均衡转发、不做响应缓存"**——apiserver 请求大多带强一致性语义（Watch、resourceVersion、写操作），不能像 DNS 那样缓存旧值，所以这里只做转发不缓存。

在非托管、自建的 Kubernetes 高可用集群中，工作节点需要通过 APIServer 与控制平面通信。为了实现对多台 Master 节点的高可用访问，除了使用外部硬件负载均衡器（如 F5、云厂商 SLB）或基于 VIP 的方案（如 Keepalived、kube-vip）外，业界还有一种**去中心化的本地代理（Local Proxy）**架构。这种方案在 **Kubespray**（作为默认 HA 实现，名为 `nginx-proxy`）以及 **Rancher RKE/RKE2**（控制平面组件同样经本地代理）中被采用。Kubeadm 自身则推荐外部 LB 或 kube-vip，不内置此方案。

### 核心架构图

```text
┌────────────────────────────────────────────────────────┐
│                      Worker Node                       │
│                                                        │
│  ┌──────────┐     ┌───────────────────────┐            │
│  │ kubelet  │     │ nginx-proxy (Static)  │            │
│  │          │────▶│ bind: 127.0.0.1:6443  │            │
│  └──────────┘     │ tcp stream loadbalance│            │
│                   └──────────┬────────────┘            │
│  ┌──────────┐                │                         │
│  │kube-proxy│────────────────┤                         │
│  └──────────┘                │                         │
└──────────────────────────────┼─────────────────────────┘
                               │
           ┌───────────────────┼───────────────────┐
           ▼                   ▼                   ▼
    ┌────────────┐      ┌────────────┐      ┌────────────┐
    │  Master 1  │      │  Master 2  │      │  Master 3  │
    │ APIServer  │      │ APIServer  │      │ APIServer  │
    └────────────┘      └────────────┘      └────────────┘
```

### 具体实现细节与配置示例

要实现这套方案，核心是在每个 Worker 节点的 `/etc/kubernetes/manifests` 目录下放置一个代理 Pod 的 yaml，同时提供对应的配置文件。`kubelet` 启动时会自动监控该目录并拉起这些静态 Pod。

#### 1. 静态 Pod 配置 (`kube-nginx.yaml`)

这是一个典型的放置在 `/etc/kubernetes/manifests/kube-nginx.yaml` 中的配置。它定义了一个直接挂载了宿主机网络的 Nginx 容器：

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: nginx-proxy
  namespace: kube-system
  labels:
    app: kube-nginx
spec:
  # 关键点 1：必须使用 HostNetwork，直接监听宿主机本地端口
  hostNetwork: true
  # 关键点 2：设为集群最高优先级，防止被 Kubelet 资源不足时驱逐
  priorityClassName: system-node-critical
  containers:
  - name: nginx-proxy
    image: nginx:alpine
    volumeMounts:
    - mountPath: /etc/nginx
      name: etc-nginx
      readOnly: true
  volumes:
  - hostPath:
      path: /etc/kubernetes/nginx-proxy
    name: etc-nginx
```

#### 2. Nginx 反向代理配置 (`nginx.conf`)

上述 Pod 挂载了宿主机的 `/etc/kubernetes/nginx-proxy` 目录。我们在这个目录下提供真实的 `nginx.conf`，利用 Nginx 的四层（TCP）转发能力：

```nginx
# /etc/kubernetes/nginx-proxy/nginx.conf
error_log stderr notice;
worker_processes 1;
events {
  multi_accept on;
  use epoll;
  worker_connections 1024;
}

# 核心部分：四层 TCP 负载均衡
stream {
    upstream kube_apiserver {
        least_conn; # 最少连接数轮询
        # 写入真实 Master 节点集群的 IP 和 端口
        server 10.20.0.7:6443;
        server 10.20.0.8:6443;
        server 10.20.0.9:6443;
    }

    server {
        # 在 Worker 节点的 127.0.0.1 监听 6443 端口
        listen        127.0.0.1:6443; 
        proxy_pass    kube_apiserver;
        proxy_timeout 10m;
        proxy_connect_timeout 10s;
    }
}
```

#### 3. kubelet 通信配置 (`kubeconfig`)

当 Nginx 在后台跑起来后，我们只需要让该节点上的 `kubelet` 和 `kube-proxy` 往本地去连即可。通过修改它们的 kubeconfig 文件：

```yaml
# /etc/kubernetes/kubelet.conf (节选)
apiVersion: v1
clusters:
- cluster:
    # 核心：将服务器地址指向本地刚起好的代理监听端口
    server: https://127.0.0.1:6443 
    certificate-authority-data: ...
```

### 通信链路解析

在这套配置下，节点网络中存在两种截然不同的 APIServer 访问链路：

1. **节点组件链路**：
   Worker 节点上的核心组件（`kubelet`、`kube-proxy`）发出的请求：
   `kubelet` -> `https://127.0.0.1:6443` -> 被本地 Nginx 接收 -> 根据 least_conn 策略进行 TCP 转发 -> `真正的 Master 节点 IP (如 10.20.0.8:6443)`。
2. **普通业务 Pod 链路（极其重要）**：
   业务 Pod **不走** 此本地 Nginx 代理。业务 Pod 访问 APIServer 时，调用的是内置服务域名 `kubernetes.default.svc.cluster.local`（对应 ClusterIP，如 `10.96.0.1:443`）。流量会被本机的 `kube-proxy` 写入的 iptables/IPVS 规则直接拦截并 NAT 转发到后端真实的 Master 节点。

### 优缺点分析

* **优点**：
  * **零额外成本与去中心化**：无需购买公有云 LB 或额外部署 Keepalived 节点。
  * **容灾能力极强**：如果某台 Master 宕机，Worker 节点本地代理会自动将其剔除，kubelet 的请求会瞬间平滑切换，避免单点故障。
  * **规避网络协议限制**：传统 Keepalived 需要 ARP 协议支持 VIP 漂移，在很多公有云 VPC 网络中默认被拦截，而本地代理基于纯 TCP/IP，不受此限制。
* **缺点**：
  * **连接数膨胀**：每个 Worker 都与所有 Master 建立直连，Nginx upstream 连接数 ≈ Worker × Master。数千节点时，Master 侧 apiserver 的连接压力显著上升。
  * **故障感知慢（被动健康检查）**：nginx 开源版 stream upstream 仅支持**被动**健康检查——请求失败才标记 down，不会主动发探针（主动 `health_check` 是 nginx Plus 商业版功能）。Master 宕机后要等首个请求失败才剔除，需靠 `max_fails`/`fail_timeout` 调优缓解。
  * **节点维护成本**：扩缩容 Master 节点时，需要同步更新所有 Worker 节点上的代理配置文件。

## 本章小结

| 组件 | 一句话总结 | 端口 | 关键概念 |
|------|-----------|------|---------|
| kubelet | 节点代理，管理 Pod 全生命周期 | TCP 10250, 10248 | 探针机制, 驱逐策略, QoS 等级 |
| kube-proxy | 网络规则管理，实现 Service 抽象 | TCP 10249, 10256 | iptables/IPVS, DNAT, 负载均衡 |
| 容器运行时 | 底层容器引擎 | Unix Socket（CRI gRPC） | CRI 接口, containerd, shim 机制 |
| nginx-proxy（可选 HA） | per-node 本地 LB，转发到多 Master apiserver | TCP 127.0.0.1:6443 | 四层负载均衡, HA 去中心化, 仅转发不缓存 |

### 节点管理端口

| 协议 | 端口 | 说明 |
|------|------|------|
| **TCP** | **22** | SSH 远程登录端口。用于节点运维管理（kubeadm 初始化、日志排查、手动修复等），与 K8s 组件无关但对集群运维不可或缺 |
| **Unix Socket** | `/run/containerd/containerd.sock` | containerd 的 CRI gRPC 接口。kubelet 通过此 Socket 与容器运行时通信（不走网络端口） |
