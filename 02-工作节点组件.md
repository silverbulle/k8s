# 02 - 工作节点组件

工作节点（Worker Node）是 K8s 集群中真正运行工作负载的机器。每个 Worker 节点运行三个核心组件：kubelet、kube-proxy 和容器运行时。

```
┌────────────────────────────────────────────────────┐
│                    Worker Node                      │
│                                                    │
│  ┌──────────┐   ┌──────────┐   ┌───────────────┐  │
│  │ kubelet  │   │kube-proxy│   │ Container RT  │  │
│  │          │   │          │   │ (containerd)  │  │
│  │ Pod 生命 │   │ 网络规则 │   │               │  │
│  │ 周期管理 │   │ 管理     │   │ 运行容器      │  │
│  └──────────┘   └──────────┘   └───────────────┘  │
│                                                    │
│  ┌─────────┐ ┌─────────┐ ┌─────────┐             │
│  │  Pod A  │ │  Pod B  │ │  Pod C  │             │
│  └─────────┘ └─────────┘ └─────────┘             │
└────────────────────────────────────────────────────┘
```

## 2.1 kubelet

kubelet 是每个 Worker 节点上的"节点代理"，负责 Pod 的生命周期管理。它从 apiserver 接收 PodSpec，确保容器按照期望状态运行。

### 核心职责

| 职责 | 说明 |
|------|------|
| **Pod 生命周期** | 创建/启动/停止/删除容器 |
| **健康检查** | 执行 Liveness/Readiness/Startup Probe |
| **状态上报** | 定期向 apiserver 汇报 Pod 和 Node 状态 |
| **Volume 管理** | 挂载/卸载 Pod 所需的 Volume |
| **镜像拉取** | 通过容器运行时拉取镜像 |
| **日志与 exec** | 提供容器日志查询和 exec 入口 |

### kubelet 如何获取 Pod 配置

kubelet 通过三种方式获取 Pod 配置（PodSpec）：

1. **API Server（主要方式）**：Watch `/api/v1/pods`，获取调度到本节点的 Pod
2. **Static Pod**：读取本地 manifest 目录（默认 `/etc/kubernetes/manifests/`），kube-apiserver 等控制平面组件就是这种方式运行的
3. **HTTP URL（已废弃）**：从 URL 定期拉取 Pod 配置

### Static Pod — 静态 Pod

上面第 2 条提到的 Static Pod 值得展开。**它不是一种"工作负载资源"**（Deployment/StatefulSet 那套有控制器、有 replicas/selector 的体系），而是一种 **Pod 的创建来源**——由 **kubelet 直接从节点本地 manifest 文件创建**，不经 apiserver。

| 维度 | 工作负载资源（Deployment/StatefulSet/...） | Static Pod |
|------|----------------------------------------|-----------|
| 谁创建 Pod | 控制器 watch apiserver 后创建 | **kubelet 直接读本地 manifest 文件创建** |
| 经 apiserver 创建吗 | 经（用户提交 yaml → apiserver → controller） | **不经 apiserver 创建** |
| yaml 存哪 | 集群 etcd 里 | 节点本地目录（默认 `/etc/kubernetes/manifests/`） |
| 谁管生命周期 | 对应 controller | **本节点 kubelet**，挂了自动重新拉起 |
| apiserver 能控制吗 | 能（可 kubectl scale/delete） | **不能**，apiserver 里只是只读镜像 |

#### 只读镜像机制

kubelet 把 static pod 起起来后，会**反向同步**一个 Pod 对象到 apiserver，所以 `kubectl get pods -n kube-system` 能看到它们。但这个对象是**只读镜像**——apiserver 不能真正控制它：

```bash
# 看到 kube-system 里带这个注解的都是 static pod
kubectl get pod etcd-master -n kube-system -o jsonpath='{.metadata.annotations}'
# 里面有 "kubernetes.io/config.source": "file"

# 尝试 kubectl delete 会被 kubelet 立即重新拉起，删不掉
kubectl delete pod etcd-master -n kube-system
# → 删除后几秒内又自动重建
```

**真正删除/修改 static pod 的唯一方式是操作节点上的 manifest 文件**——删文件则 kubelet 自动清理对应 Pod，改文件则 kubelet 重新加载。

#### 典型用途

最重要的用途是**自建集群的控制平面组件**。kubeadm 集群里 `kube-apiserver`、`etcd`、`kube-scheduler`、`kube-controller-manager` 都是 static pod：

```bash
# 查看控制平面 manifest 文件
ls /etc/kubernetes/manifests/
# etcd.yaml  kube-apiserver.yaml  kube-controller-manager.yaml  kube-scheduler.yaml
```

为什么控制平面组件必须用 static pod？因为 **apiserver 自己还没起来时，不能用 apiserver 管理自己**——static pod 只依赖 kubelet 和本地文件，能在 apiserver 不可用时仍把控制平面拉起来，这是 bootstrap 的关键。节点本地的监控 agent、日志 agent 也常用 static pod（避免 agent 挂了没人拉起）。

#### 与 DaemonSet 的区别（易混）

两者都能实现"每个节点跑一个"，但层次不同：

| | DaemonSet | Static Pod |
|--|-----------|-----------|
| 经 apiserver | ✅ | ❌ |
| 有 controller | ✅（daemonset controller） | ❌ |
| 跨所有节点 | ✅ 自动 | 仅 manifest 文件所在节点 |
| kubectl 管理 | ✅ | ❌（要登节点改文件） |
| 依赖 apiserver 存活 | ✅ | ❌ |

**选择原则**：需要 apiserver 集中管理、滚动更新 → DaemonSet；需要不依赖 apiserver、节点自举（如控制平面自身）→ Static Pod。

### 探针机制（Probes）

kubelet 通过探针监控容器的健康状态：

```yaml
containers:
- name: app
  image: myapp:1.0
  
  # Startup Probe - 容器启动期间检测（成功后其他探针才生效）
  startupProbe:
    httpGet:
      path: /healthz
      port: 8080
    failureThreshold: 30       # 最多允许 30 次失败
    periodSeconds: 10          # 每 10 秒检查一次 = 最多等 5 分钟启动

  # Liveness Probe - 判断容器是否需要重启
  livenessProbe:
    httpGet:
      path: /healthz
      port: 8080
    initialDelaySeconds: 5
    periodSeconds: 10
    failureThreshold: 3        # 连续 3 次失败 → 重启容器

  # Readiness Probe - 判断容器是否可以接收流量
  readinessProbe:
    httpGet:
      path: /ready
      port: 8080
    periodSeconds: 5
    failureThreshold: 3        # 连续 3 次失败 → 从 Service Endpoints 移除
```

**三种探针对比：**

| 探针类型 | 失败动作 | 典型用途 |
|---------|---------|---------|
| Liveness | 重启容器 | 检测死锁、无限循环 |
| Readiness | 从 Service 摘除 | 应用预热、依赖就绪检测 |
| Startup | 停止其他探针直到成功 | 慢启动应用（Java 等） |

**探针支持四种检测方式：**

```yaml
# 1. HTTP GET
httpGet:
  path: /healthz
  port: 8080
  httpHeaders:
  - name: Custom-Header
    value: value

# 2. TCP Socket
tcpSocket:
  port: 3306

# 3. Exec 命令
exec:
  command:
  - cat
  - /tmp/healthy

# 4. gRPC（1.24+）
grpc:
  port: 50051
```

### Pod 生命周期阶段

```
Pending → Running → Succeeded
                  → Failed

Pending:   调度中或镜像拉取中
Running:   至少一个容器在运行
Succeeded: 所有容器正常退出（Job 类 Pod）
Failed:    至少一个容器异常退出
Unknown:   无法获取 Pod 状态（通常节点故障）
```

### 节点状态与驱逐

kubelet 监控节点资源压力，触发驱逐（Eviction）：

```bash
# 查看节点状态
kubectl describe node <node-name>

# Conditions 说明
Ready:              True   # 节点健康，可以接受 Pod
MemoryPressure:     False  # 内存不足
DiskPressure:       False  # 磁盘不足
PIDPressure:        False  # 进程数过多
NetworkUnavailable: False  # 网络未配置
```

**驱逐优先级**（当资源压力增大时）：

1. 先驱逐 BestEffort QoS 的 Pod
2. 再驱逐 Burstable QoS 的 Pod
3. 最后驱逐 Guaranteed QoS 的 Pod

### kubelet 配置

```yaml
# kubelet-config.yaml（常用配置项）
apiVersion: kubelet.config.k8s.io/v1beta1
kind: KubeletConfiguration

# 资源预留
kubeReserved:          # K8s 系统组件预留
  cpu: "100m"
  memory: "256Mi"
systemReserved:        # 操作系统预留
  cpu: "100m"
  memory: "256Mi"

# 驱逐阈值
evictionHard:
  memory.available: "100Mi"
  nodefs.available: "10%"
  imagefs.available: "15%"

# 镜像 GC
imageGCHighThresholdPercent: 85
imageGCLowThresholdPercent: 80

# 最大 Pod 数（默认 110）
maxPods: 110
```

### 端口

| 协议 | 端口 | 说明 |
|------|------|------|
| **TCP** | **10250** | kubelet HTTPS 端口。提供 metrics 指标、exec 进入容器（`kubectl exec` 底层调用此端口）、logs 查询等 API。apiserver 通过此端口与 kubelet 通信 |
| **TCP** | **10248** | kubelet 本地健康检查端口（`/healthz`）。默认仅绑定 `localhost`，供本机探针检测 kubelet 是否存活 |

> 10250 是集群内最敏感的端口之一——拥有此端口访问权限等同于可以在任意 Node 上执行命令。务必通过 RBAC 和 mTLS 保护。

## 2.2 kube-proxy

kube-proxy 在每个 Worker 节点上维护网络规则，实现 **Service 抽象**——让 Pod 间通信不依赖具体 IP。

### 工作模式

| 模式 | 实现方式 | 适用场景 |
|------|---------|---------|
| **iptables**（默认） | 每个 Service 对应一组 iptables 规则 | 中小规模集群（< 1000 Service） |
| **IPVS** | Linux IPVS 内核模块，哈希表查找 | 大规模集群（> 1000 Service） |
| **nftables**（1.29+） | nftables 替代 iptables | 新一代 Linux 防火墙 |

### iptables 模式原理

当创建 Service 时，kube-proxy 在本节点写入 iptables 规则：

```
# 访问 Service ClusterIP → DNAT 到后端 Pod IP
-A KUBE-SERVICES -d 10.96.0.10/32 -p tcp --dport 53 -j KUBE-SVC-XXXXX
-A KUBE-SVC-XXXXX -m statistic --mode random --probability 0.33 -j KUBE-SEP-1  # 1/3 → Pod A
-A KUBE-SVC-XXXXX -m statistic --mode random --probability 0.50 -j KUBE-SEP-2  # 1/2 → Pod B
-A KUBE-SVC-XXXXX -j KUBE-SEP-3                                                 # 1/1 → Pod C
```

**关键理解：** Service 的 ClusterIP 是一个虚拟 IP，不绑定任何网络接口。流量通过 iptables DNAT 直接转发到后端 Pod，不经过中间代理。

### IPVS 模式原理

```bash
# IPVS 虚拟服务
ipvsadm -ln
IP Virtual Server version 1.2.1
Prot LocalAddress:Port Scheduler Flags
  -> RemoteAddress:Port           Forward Weight ActiveConn InActConn
TCP  10.96.0.10:53 rr             # ClusterIP:53 → round-robin
  -> 10.244.1.5:53                Masq    1      0          0
  -> 10.244.2.8:53                Masq    1      0          0
  -> 10.244.3.2:53                Masq    1      0          0
```

IPVS 支持多种负载均衡算法：rr（轮询）、wrr（加权轮询）、lc（最少连接）、wlc（加权最少连接）、sh（源哈希）、sed（最短期望延迟）、nq（不排队调度）。

### kube-proxy 配置

```yaml
apiVersion: kubeproxy.config.k8s.io/v1alpha1
kind: KubeProxyConfiguration
mode: "ipvs"              # 切换为 IPVS 模式

ipvs:
  scheduler: "rr"         # 负载均衡算法
  minSyncPeriod: "0s"
  syncPeriod: "30s"

# conntrack 配置
conntrack:
  maxPerCore: 32768       # 每个 CPU 核心最大连接跟踪数
  min: 131072
  tcpCloseWaitTimeout: "1h0m0s"
```

### 端口

| 协议 | 端口 | 说明 |
|------|------|------|
| **TCP** | **10249** | kube-proxy metrics 端口（`/metrics`）。默认仅绑定 `localhost`，暴露 Prometheus 格式的指标供本地采集 |
| **TCP** | **10256** | kube-proxy 健康检查端口（`/healthz`）。可供外部负载均衡器或监控系统探测 kube-proxy 自身是否健康 |

> IPVS 模式下，kube-proxy 还会在节点上创建 LVS 规则，并通过 10256 端口探测每个 endpoint 是否健康来决定是否继续向其转发流量。

## 2.3 容器运行时

容器运行时（Container Runtime）是真正负责运行容器的底层组件。K8s 通过 **CRI（Container Runtime Interface）** 与运行时交互。

### CRI 架构

```
kubelet
  │
  │ gRPC (CRI 接口)
  ▼
┌─────────────────────────┐
│   Container Runtime     │
│                         │
│  RuntimeService:        │
│    RunPodSandbox()      │
│    CreateContainer()    │
│    StartContainer()     │
│    StopContainer()      │
│    RemoveContainer()    │
│                         │
│  ImageService:          │
│    PullImage()          │
│    ListImages()         │
│    RemoveImage()        │
└─────────────────────────┘
```

### 主流运行时对比

| 运行时 | 维护方 | 特点 |
|--------|--------|------|
| **containerd** | CNCF | 轻量、稳定，目前主流选择 |
| **CRI-O** | Red Hat | 专为 K8s 设计，最小化攻击面 |
| **Docker Engine** | Docker | 通过 cri-dockerd 适配器支持 CRI |

### containerd 架构

```
containerd
  ├── shim（每个 Pod 一个）
  │   └── runc（创建容器）
  │       └── 容器进程
  ├── shim
  │   └── runc
  │       └── 容器进程
  └── ...
```

**shim 进程的作用：** containerd 重启时，shim 保持容器运行不中断，实现守护进程的无状态化。

### 镜像拉取流程

```
kubelet 收到 PodSpec（含镜像名）
  │
  ├─ 本地有镜像？──是──→ 直接使用
  │
  └─ 没有？──→ imagePullSecrets 认证
               │
               └─→ CRI PullImage() → containerd/docker pull
                    │
                    └─→ 从 Registry 拉取镜像层
```

**imagePullPolicy 策略：**

| 策略 | 行为 |
|------|------|
| `Always` | 每次都拉取（`latest` 标签默认值） |
| `IfNotPresent` | 本地没有才拉取（指定版本标签默认值） |
| `Never` | 只使用本地镜像 |

### CRI 与 GPU：运行时分层

前面讲的是通用容器。当涉及 **GPU** 时，有个关键认知：**CRI 接口本身根本不碰 GPU，GPU 的接入是绕过 CRI、靠运行时分层在 containerd 之下注入的。**

#### K8s 与 containerd 的解耦关系

K8s 自己不运行容器，只编排；真正运行容器的是 containerd，两者通过 **CRI（Container Runtime Interface）** 这个 gRPC 接口解耦：

```
kubelet
  │ gRPC（CRI 接口）
  ▼
containerd ── 管理镜像、容器生命周期、shim
  │
  ▼
runc ── 按 OCI 规范真正创建/启动容器进程
```

**为什么是 containerd 不是 docker：** docker 太重（含构建、CLI、卷等）且不原生支持 CRI（要靠 cri-dockerd 适配），containerd 轻量、CNCF 项目、原生实现 CRI。K8s 1.24 移除 dockershim 后，containerd 成事实标准（即 docker 架构已过时）。

**CRI 接口两大类：**
- **RuntimeService**：RunPodSandbox / CreateContainer / StartContainer / StopContainer / Exec / Logs——容器生命周期
- **ImageService**：PullImage / ListImages / RemoveImage——镜像管理

对 K8s 来说，底下是 containerd 还是 CRI-O 都一样，只要实现 CRI 接口即可——这就是 CRI 解耦的意义。

#### 关键认知：CRI 不碰 GPU

CRI 接口里**没有任何 GPU 相关方法**。kubelet 调 `CreateContainer()` 时传的是标准字段（镜像、命令、资源 requests/limits、挂载），CRI 不知道什么是 GPU。GPU 怎么进容器？靠**运行时分层 + RuntimeClass 切换**，在 containerd 之下注入：

```
通用容器（无 GPU）              GPU 容器
kubelet → CRI → containerd      kubelet → CRI → containerd
              ↓                                  ↓ 识别 RuntimeClass: nvidia
         → runc（标准 OCI）        → nvidia-container-runtime（替代 runc）
                                          ↓ 注入 GPU 设备/库
                                        runc（最终仍用 runc 起进程）
```

要点：
1. **kubelet 和 CRI 这层完全不变**——照样调 `CreateContainer()`，只是 containerd 根据 RuntimeClass 选择底层 runtime handler
2. GPU 的"魔法"发生在 **containerd 之下**：把默认 `runc` 换成 `nvidia-container-runtime`
3. nvidia-container-runtime 最终仍调 runc 创建容器，只是在调用前**预处理**：把 GPU 设备节点（/dev/nvidia*）、CUDA 库、环境变量注入容器

#### GPU 容器创建链路（CRI 视角）

```
1. Pod 声明 resources.limits: { nvidia.com/gpu: 1 } + runtimeClassName: nvidia

2. kubelet → CRI CreateContainerRequest（带 runtimeHandler="nvidia"）
   ★ nvidia.com/gpu 是 Extended Resource，调度器已保证节点有配额；
     CRI 本身不解析它，只把 runtimeHandler 传给 containerd

3. containerd 收到请求，看 runtimeHandler="nvidia"
   → 查 config.toml 找到 [plugins.cri.containerd.runtimes.nvidia]
   → 该 handler 的 BinaryName = /usr/bin/nvidia-container-runtime

4. nvidia-container-runtime 接管，做 GPU 注入预处理：
   - 解析容器注解 NVIDIA_VISIBLE_DEVICES（决定用哪些 GPU）
   - 把 /dev/nvidia0 等设备节点挂进容器 namespace
   - 把 CUDA 库（libcuda.so 等）挂进容器
   - 设置环境变量
   ★ 依赖 nvidia-container-toolkit（原 NVIDIA Container Toolkit）

5. nvidia-container-runtime → runc → 创建容器进程（进程内可见 GPU）
```

**两个关键配置：**

RuntimeClass（K8s 层，告诉 kubelet 用哪个 handler）：
```yaml
apiVersion: node.k8s.io/v1
kind: RuntimeClass
metadata: { name: nvidia }
handler: nvidia          # 对应 containerd 的 runtime handler 名
```

containerd config.toml（containerd 层，定义 handler 指向哪个 runtime）：
```toml
[plugins."io.containerd.grpc.v1.cri".containerd.runtimes.nvidia]
  runtime_type = "io.containerd.runc.v2"
  [plugins."io.containerd.grpc.v1.cri".containerd.runtimes.nvidia.options]
    BinaryName = "/usr/bin/nvidia-container-runtime"
```

#### 演进：CDI（Container Device Interface）

老方式（nvidia-container-runtime 注入）的局限：GPU 注入逻辑绑死在 NVIDIA runtime 里，其它设备（AMD GPU、RDMA 网卡、FPGA）各搞各的，不统一。

**CDI 是新标准**（K8s 1.28+ GA）：让设备厂商用统一格式声明"这个设备怎么暴露给容器"，containerd 原生支持 CDI，不再需要每家一个专用 runtime。意义是**把"设备注入"从 runtime 层上移到标准化设备描述**，nvidia/AMD/RDMA 都走同一套机制——这是 GPU + CRI 的未来方向，GPU Operator 新版已支持 CDI 模式。

> 更完整的 GPU 链路（GPU Operator、GPU 调度、NCCL 集合通信等）见 [05b-GPU与高性能互联](05b-GPU与高性能互联.md)。本节只聚焦"CRI 这层如何处理 GPU"。

## 2.4 APIServer 高可用本地代理 (Local Proxy)

> 说明：这是一个**可选的 HA 实践**，并非工作节点的核心组件（核心仍是 kubelet、kube-proxy、容器运行时三件套）。它解决的是"工作节点如何高可用地访问多 Master apiserver"这一问题，与 [4.4 NodeLocal DNSCache](04-服务与网络.md#nodelocal-dnscache--dns-性能优化) 思路相近但有本质区别：**DNSCache 是"本地缓存命中即返回"，而本节方案是"仅负载均衡转发、不做响应缓存"**——apiserver 请求大多带强一致性语义（Watch、resourceVersion、写操作），不能像 DNS 那样缓存旧值，所以这里只做转发不缓存。

在非托管、自建的 Kubernetes 高可用集群中，工作节点需要通过 APIServer 与控制平面通信。为了实现对多台 Master 节点的高可用访问，除了使用外部硬件负载均衡器（如 F5、云厂商 SLB）或基于 VIP 的方案（如 Keepalived、kube-vip）外，业界还有一种**去中心化的本地代理（Local Proxy）**架构。这种方案在 **Kubespray**（作为默认 HA 实现，名为 `nginx-proxy`）以及 **Rancher RKE/RKE2**（控制平面组件同样经本地代理）中被采用。Kubeadm 自身则推荐外部 LB 或 kube-vip，不内置此方案。

### 核心架构图

```text
┌────────────────────────────────────────────────────────┐
│                      Worker Node                       │
│                                                        │
│  ┌──────────┐     ┌───────────────────────┐            │
│  │ kubelet  │     │ nginx-proxy (Static)  │            │
│  │          │────▶│ bind: 127.0.0.1:6443  │            │
│  └──────────┘     │ tcp stream loadbalance│            │
│                   └──────────┬────────────┘            │
│  ┌──────────┐                │                         │
│  │kube-proxy│────────────────┤                         │
│  └──────────┘                │                         │
└──────────────────────────────┼─────────────────────────┘
                               │
           ┌───────────────────┼───────────────────┐
           ▼                   ▼                   ▼
    ┌────────────┐      ┌────────────┐      ┌────────────┐
    │  Master 1  │      │  Master 2  │      │  Master 3  │
    │ APIServer  │      │ APIServer  │      │ APIServer  │
    └────────────┘      └────────────┘      └────────────┘
```

### 具体实现细节与配置示例

要实现这套方案，核心是在每个 Worker 节点的 `/etc/kubernetes/manifests` 目录下放置一个代理 Pod 的 yaml，同时提供对应的配置文件。`kubelet` 启动时会自动监控该目录并拉起这些静态 Pod。

#### 1. 静态 Pod 配置 (`kube-nginx.yaml`)

这是一个典型的放置在 `/etc/kubernetes/manifests/kube-nginx.yaml` 中的配置。它定义了一个直接挂载了宿主机网络的 Nginx 容器：

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: nginx-proxy
  namespace: kube-system
  labels:
    app: kube-nginx
spec:
  # 关键点 1：必须使用 HostNetwork，直接监听宿主机本地端口
  hostNetwork: true
  # 关键点 2：设为集群最高优先级，防止被 Kubelet 资源不足时驱逐
  priorityClassName: system-node-critical
  containers:
  - name: nginx-proxy
    image: nginx:alpine
    volumeMounts:
    - mountPath: /etc/nginx
      name: etc-nginx
      readOnly: true
  volumes:
  - hostPath:
      path: /etc/kubernetes/nginx-proxy
    name: etc-nginx
```

#### 2. Nginx 反向代理配置 (`nginx.conf`)

上述 Pod 挂载了宿主机的 `/etc/kubernetes/nginx-proxy` 目录。我们在这个目录下提供真实的 `nginx.conf`，利用 Nginx 的四层（TCP）转发能力：

```nginx
# /etc/kubernetes/nginx-proxy/nginx.conf
error_log stderr notice;
worker_processes 1;
events {
  multi_accept on;
  use epoll;
  worker_connections 1024;
}

# 核心部分：四层 TCP 负载均衡
stream {
    upstream kube_apiserver {
        least_conn; # 最少连接数轮询
        # 写入真实 Master 节点集群的 IP 和 端口
        server 10.20.0.7:6443;
        server 10.20.0.8:6443;
        server 10.20.0.9:6443;
    }

    server {
        # 在 Worker 节点的 127.0.0.1 监听 6443 端口
        listen        127.0.0.1:6443; 
        proxy_pass    kube_apiserver;
        proxy_timeout 10m;
        proxy_connect_timeout 10s;
    }
}
```

#### 3. kubelet 通信配置 (`kubeconfig`)

当 Nginx 在后台跑起来后，我们只需要让该节点上的 `kubelet` 和 `kube-proxy` 往本地去连即可。通过修改它们的 kubeconfig 文件：

```yaml
# /etc/kubernetes/kubelet.conf (节选)
apiVersion: v1
clusters:
- cluster:
    # 核心：将服务器地址指向本地刚起好的代理监听端口
    server: https://127.0.0.1:6443 
    certificate-authority-data: ...
```

### 通信链路解析

在这套配置下，节点网络中存在两种截然不同的 APIServer 访问链路：

1. **节点组件链路**：
   Worker 节点上的核心组件（`kubelet`、`kube-proxy`）发出的请求：
   `kubelet` -> `https://127.0.0.1:6443` -> 被本地 Nginx 接收 -> 根据 least_conn 策略进行 TCP 转发 -> `真正的 Master 节点 IP (如 10.20.0.8:6443)`。
2. **普通业务 Pod 链路（极其重要）**：
   业务 Pod **不走** 此本地 Nginx 代理。业务 Pod 访问 APIServer 时，调用的是内置服务域名 `kubernetes.default.svc.cluster.local`（对应 ClusterIP，如 `10.96.0.1:443`）。流量会被本机的 `kube-proxy` 写入的 iptables/IPVS 规则直接拦截并 NAT 转发到后端真实的 Master 节点。

### 优缺点分析

* **优点**：
  * **零额外成本与去中心化**：无需购买公有云 LB 或额外部署 Keepalived 节点。
  * **容灾能力极强**：如果某台 Master 宕机，Worker 节点本地代理会自动将其剔除，kubelet 的请求会瞬间平滑切换，避免单点故障。
  * **规避网络协议限制**：传统 Keepalived 需要 ARP 协议支持 VIP 漂移，在很多公有云 VPC 网络中默认被拦截，而本地代理基于纯 TCP/IP，不受此限制。
* **缺点**：
  * **连接数膨胀**：每个 Worker 都与所有 Master 建立直连，Nginx upstream 连接数 ≈ Worker × Master。数千节点时，Master 侧 apiserver 的连接压力显著上升。
  * **故障感知慢（被动健康检查）**：nginx 开源版 stream upstream 仅支持**被动**健康检查——请求失败才标记 down，不会主动发探针（主动 `health_check` 是 nginx Plus 商业版功能）。Master 宕机后要等首个请求失败才剔除，需靠 `max_fails`/`fail_timeout` 调优缓解。
  * **节点维护成本**：扩缩容 Master 节点时，需要同步更新所有 Worker 节点上的代理配置文件。

## 2.5 节点加入与就绪流程

前面分组件讲了 kubelet、kube-proxy、容器运行时，但一个**空白节点从加入集群到 Ready** 的完整链路还没串起来。这一节用 **2.4 的 nginx 本地代理方案**为主线把流程走一遍——因为 nginx-proxy 本身就是个 **static pod**，正好把 static pod、CNI、kube-proxy、DaemonSet 调度串成一条时间线。

> **为什么拿 nginx 代理方案讲？** 因为这个方案下 Worker 节点的 manifest 目录非空——里面躺着 `kube-nginx.yaml`，节点就绪流程里"读 manifest 拉 static pod"这一步就有了具体对象。而直连方案（见末尾对比）下 Worker 的 manifest 目录为空，这一步无 static pod 可拉。用 nginx 方案讲能完整覆盖 static pod 环节，更有代表性。

### 阶段一：节点引导加入集群

核心是**建立 kubelet 与 apiserver 的互信**，让空白节点变成集群成员：

```
1. 控制平面 kubeadm init → 生成 join token（带 TTL）和 CA 证书指纹

2. 新节点 kubeadm join <master>:6443 --token <token> --discovery-token-ca-cert-hash <hash>
   a. 用 token 通过 bootstrap 认证，下载集群 CA 证书
   b. TLS Bootstrap：先用 token 换临时凭证，kubelet 再用临时凭证
      向 apiserver 申请正式客户端证书
   c. 写入 kubelet.conf（kubeconfig）
   d. 启动 kubelet（systemd 服务，先于一切 K8s Pod 组件）

3. kubelet 持 kubelet.conf 连 apiserver → 注册为 Node
```

> **关键点：kubelet 是第一个起来的 K8s 组件，而且是 systemd 服务（`systemctl start kubelet`），不是 Pod。** Pod 要靠它拉起，所以它必须先于 Pod 存在。此时还没有 calico-node、kube-proxy、nginx-proxy 这些 Pod——它们是 kubelet 就绪后才轮到。

### 阶段二：节点就绪后拉起本地组件（nginx 代理方案）

kubelet 起来后，**第一件事就是读本地 manifest 目录**——nginx 代理方案下，这里躺着 `kube-nginx.yaml`，于是 nginx-proxy 作为 static pod 被拉起。等它在 `127.0.0.1:6443` 监听起来，kubelet 自己的 kubeconfig 才有地方连：

```
kubelet 就绪（systemd）
  │
  ├─① 读本地 manifest /etc/kubernetes/manifests/kube-nginx.yaml
  │   → 拉起 nginx-proxy（static pod，hostNetwork）
  │   → nginx 监听 127.0.0.1:6443，转发到各 Master:6443
  │   ★ nginx 方案特有：这步把"访问 apiserver 的本地入口"建起来
  │
  ├─② kubelet 用 kubelet.conf 连 127.0.0.1:6443（经 nginx）→ 注册 Node
  │   （直连方案：kubelet.conf 直接指向 Master IP 或 LB VIP，跳过此 nginx）
  │
  ├─③ Watch apiserver：哪些 Pod 调度到了本节点？
  │
  ├─④ 拉起集群级 DaemonSet（按优先级）：
  │   - calico-node（system-node-critical）→ 建 Pod 网络
  │   - kube-proxy（system-node-critical）→ 写 Service 转发规则
  │   - node-exporter / 监控 agent（可选）
  │
  ├─⑤ 网络就绪后，拉起普通业务 Pod
  │   每个 Pod：CRI 建 sandbox → CNI 配网卡分 IP → 拉镜像 → 启容器 → 探针
  │
  └─⑥ 上报 Node Condition Ready=True → 节点正式 Ready，开始接流量
```

### 几个关键认知

**1. nginx-proxy 是 Worker 上唯一的 static pod，且必须最先起。**
nginx 代理方案下，Worker 的 manifest 目录里就一个 `kube-nginx.yaml`，kubelet 一读 manifest 就把它拉起。它优先级最高不是因为 priorityClass（static pod 不参与优先级调度），而是因为**kubelet 启动时第一件事就是扫 manifest 目录**——这步天然先于"watch apiserver 拉别的 Pod"。而且 kubelet 自己访问 apiserver 都得靠它（指向 `127.0.0.1:6443`），它不起来后续啥都干不了，形成硬依赖。

**2. calico-node、kube-proxy 不是 kubelet 主动"找来"的，而是被调度过来的。**
它们是 **DaemonSet**，DaemonSet Controller 给每个 Node 调度一个对应 Pod。kubelet 只是 **watch apiserver 发现"有 Pod 调度到我这了"** 就拉，不区分"这是 calico 还是业务 Pod"——对 kubelet 来说都是 Pod，只是优先级不同。nginx-proxy 不一样，它是 static pod，不走 DaemonSet 调度、不经 apiserver 创建，靠本地文件。

**3. calico-node 必须先于业务 Pod，靠优先级 + 运行时依赖双重保证。**
calico-node 是 `system-node-critical` 优先级，业务 Pod 通常普通优先级，资源紧张时高优先级先调度先拉起。更关键的是**运行时依赖**：业务 Pod 要分 Pod IP 必须靠 CNI（calico），calico-node 没起来 CNI 插件无法工作，业务 Pod 会卡在 `ContainerCreating`。即便调度顺序不严格，运行时也会形成"calico 先、业务后"的实际依赖。

**4. Node Ready 的标志是网络就绪，不只是 kubelet 活着。**
节点 `Ready=True` 不仅要求 kubelet 健康，还要求 CNI 能正常工作。部分集群用 `node.kubernetes.io/network-unavailable` 污点，在 CNI 就绪前挡住业务 Pod。

### nginx 代理方案 vs 直连方案对比

同样的就绪流程，两种方案只在"访问 apiserver 的入口"这一步不同：

| 环节 | nginx 代理方案 | 直连方案 |
|------|--------------|---------|
| manifest 目录 | 非空，含 `kube-nginx.yaml` | 为空（Worker 无 static pod） |
| 第①步 | 拉 nginx-proxy static pod | 无 static pod，直接进第②步 |
| kubelet.conf 的 server | `127.0.0.1:6443`（经 nginx） | Master IP（单 Master）/ LB VIP / Keepalived VIP |
| 多 Master 切换 | nginx 本地负载均衡+剔除 | LB/Keepalived 负责，或单 Master 无切换 |
| 适用 | 自建去中心化 HA，无外部 LB | 云上有 LB / 单 Master 学习环境 |

> 直连方案下，Worker 的 manifest 目录为空，"读 manifest 拉 static pod"这一步什么都不拉，kubelet 启动后直接用 kubelet.conf 里写好的 Master IP/LB VIP 连 apiserver。两种方案后续③④⑤⑥完全一样——拉 DaemonSet、拉业务 Pod、上报 Ready。

### 完整时序（nginx 代理方案）

```
时间 →

节点空白
  │ kubeadm join：拿 CA、TLS Bootstrap 换证书、写 kubelet.conf
  ▼
kubelet 起来（systemd）
  │
  ├──读 /etc/kubernetes/manifests/kube-nginx.yaml ──► nginx-proxy（static pod）
  │      监听 127.0.0.1:6443，转发各 Master
  │
  ├──kubelet.conf 连 127.0.0.1:6443（经 nginx）──► 注册 Node
  │
  ├──watch 调度到本节点的 Pod
  │     ├── calico-node（priority 最高）──► CNI 就绪，能分 Pod IP
  │     ├── kube-proxy                 ──► Service 规则就绪
  │     └── node-exporter 等
  │
  ├──拉业务 Pod ──► CRI sandbox → CNI 配网卡 → 起容器 → 探针通过
  │
  └──上报 Ready=True ──► 节点正式接流量
```

## 本章小结

| 组件 | 一句话总结 | 端口 | 关键概念 |
|------|-----------|------|---------|
| kubelet | 节点代理，管理 Pod 全生命周期 | TCP 10250, 10248 | 探针机制, 驱逐策略, QoS 等级 |
| kube-proxy | 网络规则管理，实现 Service 抽象 | TCP 10249, 10256 | iptables/IPVS, DNAT, 负载均衡 |
| 容器运行时 | 底层容器引擎 | Unix Socket（CRI gRPC） | CRI 接口, containerd, shim 机制 |
| nginx-proxy（可选 HA） | per-node 本地 LB，转发到多 Master apiserver | TCP 127.0.0.1:6443 | 四层负载均衡, HA 去中心化, 仅转发不缓存 |

### 节点管理端口

| 协议 | 端口 | 说明 |
|------|------|------|
| **TCP** | **22** | SSH 远程登录端口。用于节点运维管理（kubeadm 初始化、日志排查、手动修复等），与 K8s 组件无关但对集群运维不可或缺 |
| **Unix Socket** | `/run/containerd/containerd.sock` | containerd 的 CRI gRPC 接口。kubelet 通过此 Socket 与容器运行时通信（不走网络端口） |
