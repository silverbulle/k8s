# 01 - 控制平面组件

控制平面（Control Plane）是 K8s 集群的"大脑"，负责维护集群状态、调度决策和 API 服务。传统上运行在 Master 节点，生产环境通常为高可用部署多副本。

## 整体架构

```
                    ┌─────────────────────────────────────┐
                    │          Control Plane (Master)      │
                    │                                     │
  kubectl ────────► │  kube-apiserver ──────► etcd         │
  API 请求          │       │                             │
                    │       ├────► kube-scheduler          │
                    │       │                             │
                    │       └────► kube-controller-manager │
                    │                                     │
                    │  cloud-controller-manager (可选)     │
                    └─────────────────────────────────────┘
```

所有组件都通过 kube-apiserver 交互，**apiserver 是唯一直接访问 etcd 的组件**。

---

## 1.1 kube-apiserver

kube-apiserver 是集群的统一入口，提供 RESTful API 供所有客户端（kubectl、kubelet、controller-manager、scheduler）通信。

### 核心职责

| 职责 | 说明 |
|------|------|
| **API 服务** | 提供 RESTful 接口，CRUD 集群资源 |
| **认证（Authentication）** | 验证请求者身份（X509 证书、Bearer Token、OIDC 等） |
| **鉴权（Authorization）** | 验证请求权限（RBAC、ABAC、Webhook） |
| **准入控制（Admission）** | 在写入前拦截和修改请求（Mutating → Validating） |
| **Watch 机制** | 支持长连接监听资源变更，驱动事件驱动架构 |

### 请求处理流程

```
请求 → 认证 → 鉴权 → Mutating Admission → Schema 校验 → Validating Admission → etcd
```

**三个阶段各有明确分工：**

1. **认证**：你是谁？（X509、Token、OIDC、ServiceAccount）
2. **鉴权**：你能做什么？（RBAC 最常用，Role/ClusterRole + Binding）
3. **准入控制**：请求是否合规？可在此阶段注入 sidecar、添加标签、校验策略

### 关键特性

- **API 聚合层**：通过 APIService 注册自定义 API Server，扩展原生 API
- **CRD（Custom Resource Definition）**：无需编写代码即可定义新资源类型
- **Watch 缓存**：对 etcd 的 watch 结果做本地缓存，减少 etcd 压力
- **API Priority and Fairness**：API 限流和优先级控制，防止过载

### 常用命令

```bash
# 查看 apiserver 的 API 版本
kubectl api-versions

# 查看所有可用资源
kubectl api-resources

# 查看 RBAC 授权
kubectl get clusterroles,clusterrolebindings
```

### 端口

| 协议 | 端口 | 说明 |
|------|------|------|
| **TCP** | **6443** | kube-apiserver HTTPS 服务端口，集群所有 API 请求的唯一入口。kubectl、kubelet、controller-manager、scheduler 均通过此端口与 apiserver 通信 |

> 防火墙/安全组必须放行此端口。生产环境通常在前面挂负载均衡器（如 HAProxy、AWS NLB）。

## 1.2 etcd

etcd 是 K8s 的状态存储后端，一个高可用的分布式键值数据库，使用 Raft 共识算法保证一致性。

### 核心特点

| 特性 | 说明 |
|------|------|
| **强一致性** | 所有读操作返回最新数据（线性一致性） |
| **Watch 支持** | 监听 key 或 key 前缀的变更事件 |
| **事务操作** | 支持条件写入（If-Then-Else） |
| **TTL** | 键可设置过期时间 |
| **Raft 共识** | 多数派写入，容忍少数节点故障 |

### 数据存储结构

etcd 中所有 K8s 资源存储在 `/registry` 前缀下：

```
/registry/pods/<namespace>/<pod-name>
/registry/deployments/<namespace>/<deployment-name>
/registry/services/specs/<namespace>/<service-name>
/registry/namespaces/<namespace>
/registry/nodes/<node-name>
/registry/configmaps/<namespace>/<name>
```

数据以 protobuf 序列化的格式存储，每个资源对象包含 `metadata.resourceVersion`，用于乐观锁和 Watch 机制。

### 关键概念

- **resourceVersion**：每个资源的版本号，Watch 基于此实现增量同步
- **compact revision**：etcd 定期压缩历史版本，过老的 Watch 会收到 `Gone` 错误需要重新 List
- **Quota**：默认 2GB 存储上限，生产建议 8GB

### 端口

| 协议 | 端口 | 说明 |
|------|------|------|
| **TCP** | **2379** | etcd 客户端 API 端口。kube-apiserver 通过此端口读写 etcd 数据，是集群的关键数据路径 |
| **TCP** | **2380** | etcd 集群内部 peer 通信端口。用于 leader/follower 之间的 Raft 复制和选举 |

> 2379 只需对 apiserver 开放；2380 只需在 etcd 节点间互通，不应暴露给外部。

### 运维要点

```bash
# 查看 etcd 集群状态（需要证书）
etcdctl --endpoints=https://127.0.0.1:2379 \
  --cacert=/etc/kubernetes/pki/etcd/ca.crt \
  --cert=/etc/kubernetes/pki/etcd/server.crt \
  --key=/etc/kubernetes/pki/etcd/server.key \
  endpoint status --write-out=table

# 备份
etcdctl snapshot save /backup/etcd-snapshot.db

# 恢复
etcdctl snapshot restore /backup/etcd-snapshot.db --data-dir=/var/lib/etcd-restore
```

## 1.3 kube-scheduler

kube-scheduler 负责将未绑定节点的 Pod 分配到最合适的 Node 上。

### 调度流程

```
Pod 进入调度队列
       │
       ▼
┌──────────────┐
│  Scheduling   │  1. PreFilter → 2. Filter → 3. PostFilter
│  Cycle        │  4. Score → 5. Reserve → 6. Bind
└──────────────┘
```

**两阶段决策：**

1. **Filter（过滤）**：排除不满足约束的节点
   - 资源不足（CPU/Memory/存储）
   - 节点选择器（nodeSelector）不匹配
   - 亲和性/反亲和性（affinity/anti-affinity）不满足
   - 污点容忍（Taint/Toleration）不匹配
   - 端口冲突

2. **Score（打分）**：对通过过滤的节点评分
   - 资源均衡度（LeastRequestedPriority）
   - 节点亲和性偏好
   - 数据本地性
   - Pod 间反亲和偏好
   - 自定义打分插件

### 影响调度的关键配置

影响 Pod 落到哪个 Node 的配置分四类，按"想去哪 / 谁能来 / 怎么分布 / 谁先来"四个角度互补：

| 机制 | 角度 | 生效阶段 |
|------|------|---------|
| nodeSelector / nodeAffinity | "想去哪"(看 Node 标签) | Filter + Score |
| podAffinity / podAntiAffinity | "想去哪"(看其它 Pod 位置) | Filter + Score |
| topologySpreadConstraints | "怎么分布"(跨域均匀) | Filter + Score |
| Taints & Tolerations | "谁能来"(Node 的排斥标记) | Filter |
| PriorityClass | "谁先来 / 被抢占" | 调度队列 + 抢占 |

下面逐个展开。注意后缀 `IgnoredDuringExecution` 的含义：这些约束**只在调度时生效，运行后 Node/Pod 标签变了也不驱逐已运行 Pod**（"调度时严格、运行时宽容"）。

#### 1. 节点选择：nodeSelector 与 nodeAffinity

从粗到细三档，约束强度递增：

| 机制 | 约束强度 | 灵活度 | 典型场景 |
|------|---------|--------|---------|
| `nodeSelector` | 硬（不匹配就不调度） | 低（只能全等于） | 简单需求，如 `disk: ssd` |
| `nodeAffinity` required | 硬 | 中（多操作符） | 必须落在特定 zone/机型 |
| `nodeAffinity` preferred | 软（打分偏好，违反也能调度） | 中 | 倾向高性能节点，但不强制 |

**nodeAffinity 的 operator**：`In`（值在列表里）、`NotIn`、`Exists`（key 存在不看 value）、`DoesNotExist`、`Gt`/`Lt`（数值比较，用于资源类标签）。

**节点标签体系（nodeSelector/nodeAffinity 的基础）：**

内置标签（集群自动打，直接可用）：

| 标签 | 含义 |
|------|------|
| `kubernetes.io/hostname` | 节点名 |
| `topology.kubernetes.io/zone` | 可用区 |
| `topology.kubernetes.io/region` | 地域 |
| `kubernetes.io/os` | linux/windows |
| `kubernetes.io/arch` | amd64/arm64 |
| `node.kubernetes.io/instance-type` | 机型（云上） |

自定义标签（运维手动打，按维度划分）：

```bash
# 给节点打标签
kubectl label node node-1 disk=ssd
kubectl label node node-1 gpu=true
kubectl label node node-2 role=compute
kubectl label node node-3 env=prod

# 常见维度：disk(ssd/hdd) gpu(true) role(compute/memory/storage)
#           env(prod/staging/dev) dedicated(gpu/monitoring/ingress)
```

```yaml
# nodeSelector 实战示例
# 例1:只要 SSD 节点
nodeSelector:
  disk: ssd

# 例2:必须 GPU 节点（值用引号，布尔/数字标签易踩坑）
nodeSelector:
  gpu: "true"

# 例3:生产环境 + 计算节点（多标签 AND 关系）
nodeSelector:
  env: prod
  role: compute

# 例4:钉到指定节点（调试/排查专用）
nodeSelector:
  kubernetes.io/hostname: node-3
```

```yaml
# nodeAffinity - 更灵活的节点亲和性
affinity:
  nodeAffinity:
    requiredDuringSchedulingIgnoredDuringExecution:   # 硬性约束
      nodeSelectorTerms:
      - matchExpressions:
        - key: topology.kubernetes.io/zone
          operator: In
          values: [zone-a, zone-b]
    preferredDuringSchedulingIgnoredDuringExecution:  # 软性偏好（带权重）
    - weight: 80                       # 1-100，越大越倾向
      preference:
        matchExpressions:
        - key: node-type
          operator: In
          values: [high-performance]
```

> **边界**：nodeSelector 只能全等匹配、不支持否定/数值比较、不能表达软偏好或 OR 逻辑。需要这些能力就用 nodeAffinity。生产里 nodeSelector 常作基础硬约束，nodeAffinity preferred 补充倾向性。

#### 2. Pod 间亲和/反亲和（podAffinity / podAntiAffinity）

和节点亲和不同，它是**根据其它 Pod 的位置决定本 Pod 去哪**。关键概念是 `topologyKey`——把节点按拓扑域分组（`kubernetes.io/hostname` 按节点、`topology.kubernetes.io/zone` 按可用区）。

```yaml
affinity:
  podAntiAffinity:
    requiredDuringSchedulingIgnoredDuringExecution:   # 硬性:必须打散
    - labelSelector:
        matchLabels:
          app: web           # 不和带 app=web 的 Pod 共存
      topologyKey: kubernetes.io.hostname   # 拓扑域=单节点 → 每节点最多一个
    preferredDuringSchedulingIgnoredDuringExecution:  # 软性:尽量跨可用区
    - weight: 100
      podAffinityTerm:
        labelSelector:
          matchLabels:
            app: web
        topologyKey: topology.kubernetes.io/zone
```

- **podAntiAffinity**（反向）：把副本打散到不同节点/可用区，避免单点。最典型场景
- **podAffinity**（正向）：让 Pod 和相关 Pod 跑同一节点，如缓存 Pod 和调用者同节点降延迟

> **坑**：podAntiAffinity required + 副本数 > 拓扑域数时，多余 Pod 会**一直 Pending**（没有满足反亲和的节点）。且大规模集群下 podAntiAffinity 要遍历所有 Pod 计算开销大——这两种情况改用 topology spread constraints。

#### 3. Topology Spread Constraints — 均匀打散

podAntiAffinity 打散的局限：required 时副本超域数会 Pending，preferred 时打散程度不可控。**Topology Spread Constraints 专为"跨拓扑域均匀分布"设计**，不要求"每域一个"，而是"各域间数量差不超过阈值"。

```yaml
topologySpreadConstraints:
- maxSkew: 1                    # 各拓扑域间 Pod 数量差的最大值
  topologyKey: kubernetes.io/hostname   # 按节点分域
  whenUnsatisfiable: DoNotSchedule       # 不满足:硬性拒绝(DoNotSchedule)/软(ScheduleAnyway)
  labelSelector:
    matchLabels: { app: web }
```

| 字段 | 含义 |
|------|------|
| `maxSkew` | 各拓扑域间 Pod 数量的最大允许差值，`1` 即尽量均匀 |
| `topologyKey` | 拓扑域划分依据（hostname/zone/region） |
| `whenUnsatisfiable` | `DoNotSchedule`（硬）/ `ScheduleAnyway`（软，优先减小 skew） |
| `labelSelector` | 统计哪些 Pod 参与分布计算 |

**maxSkew 计算**：3 节点 replicas=6 `maxSkew: 1` → 理想 2/2/2；若全堆 node1 则 skew=6>1 被 `DoNotSchedule` 挡住，强制分散。允许同域多个（只要域间差≤1），比 podAntiAffinity required 更灵活。

**与 podAntiAffinity 选型：**

| 场景 | 用什么 |
|------|--------|
| 每域严格只放一个（守护进程） | podAntiAffinity required |
| 跨域均匀分布，允许同域多个 | **topology spread（maxSkew）** |
| 尽量跨可用区，不强求 | topology spread `ScheduleAnyway` 或 podAntiAffinity preferred |

生产里跨 zone/节点打散，**topology spread 现在是首选**。

#### 4. 污点与容忍（Taints & Tolerations）

污点打在 Node 上（排斥标记），容忍写在 Pod 上（免疫证明）。Node 有污点 → 默认 Pod 不往这调度，除非 Pod 声明对应容忍。

**三种 effect（区别是核心）：**

| effect | 行为 | 用途 |
|--------|------|------|
| `NoSchedule` | 调度时不往这派（已运行 Pod 不动） | 专用节点，如 GPU 节点 |
| `PreferNoSchedule` | 尽量不往这派（软，实在没地方也会派） | 尽量避开 |
| `NoExecute` | **已运行的也驱逐**（除非容忍此 effect） | 节点维护、故障清场 |

> **NoExecute 最猛**：加上后没声明对应容忍的**已运行 Pod 会被立即驱逐**。容忍 NoExecute 时可配 `tolerationSeconds`（宽限 N 秒后再走）实现优雅迁移。

```bash
# 加污点（key=dedicated value=gpu effect=NoSchedule）
kubectl taint node node-1 dedicated=gpu:NoSchedule
# 删污点（末尾加 -）
kubectl taint node node-1 dedicated=gpu:NoSchedule-
```

```yaml
tolerations:
- key: "dedicated"
  operator: "Equal"        # Equal:key+value 必须都匹配
  value: "gpu"
  effect: "NoSchedule"
- key: "node.kubernetes.io/memory-pressure"
  operator: "Exists"       # Exists:只要 key 存在即可（用于条件污点）
  effect: "NoSchedule"
```

**内置污点（系统自动打）：** `node.kubernetes.io/not-ready`、`unreachable`、`memory-pressure`、`disk-pressure`、`network-unavailable`。配合 toleration 可控制"节点故障多久后驱逐 Pod"——默认 not-ready/unreachable 300s 后 NoExecute 驱逐。

> **呼应**：上一章 2.5 节提到 CNI 就绪前用 `node.kubernetes.io/network-unavailable` 污点挡住业务 Pod——就是这个机制：节点网络未就绪时系统打上此污点，calico-node 就绪后清掉，业务 Pod 才能调度上来。

#### 小结：四类配置的协同

四类机制互补，常组合使用：用 nodeSelector/nodeAffinity 选机型 → topology spread 跨域打散 → Taint 排斥非专用 Pod → PriorityClass 保关键 Pod 优先且可抢占。它们在不同调度阶段生效（见本节开头的生效阶段表），共同决定 Pod 最终落到哪。

### PriorityClass 与抢占

```yaml
apiVersion: scheduling.k8s.io/v1
kind: PriorityClass
metadata:
  name: high-priority
value: 1000000
globalDefault: false
preemptionPolicy: PreemptLowerPriority  # 允许抢占低优先级 Pod
```

高优先级 Pod 在资源不足时可以驱逐低优先级 Pod，腾出资源。

## 1.4 kube-controller-manager

kube-controller-manager 是多个控制器的集合体，每个控制器运行独立的 **控制循环（Reconciliation Loop）**，持续将系统实际状态推向期望状态。

### 核心控制循环原理

```
        ┌──────────────────────────────────┐
        │         Controller Loop          │
        │                                  │
        │  观察当前状态 → 计算差异 → 执行动作  │
        │       ▲                    │     │
        │       └────────────────────┘     │
        │         (持续循环)                │
        └──────────────────────────────────┘
```

这个模式被称为 **"水平触发"**（level-triggered）而非"边缘触发"：控制器不关心事件本身，只关心当前状态与期望状态的差异。即使中间丢失了某些事件，最终也能收敛到正确状态。

### 主要控制器

| 控制器 | 职责 |
|--------|------|
| **Node Controller** | 监控 Node 状态，处理 Node 不可达时的 Pod 驱逐 |
| **ReplicaSet Controller** | 确保 RS 管理的 Pod 副本数与期望一致 |
| **Deployment Controller** | 管理 ReplicaSet，实现滚动更新和回滚 |
| **Endpoints Controller** | 维护 Service 与 Pod 的映射关系（Endpoints 对象） |
| **ServiceAccount Controller** | 为每个 Namespace 创建默认 ServiceAccount |
| **Namespace Controller** | 处理 Namespace 删除时的级联清理 |
| **PersistentVolume Controller** | 管理 PV 与 PVC 的绑定 |
| **Job Controller** | 管理 Job 的执行和完成状态 |
| **CronJob Controller** | 按 Cron 表达式创建 Job |

### Leader Election

kube-controller-manager 多副本部署时，通过 etcd 的 Leader Election 机制保证同一时刻只有一个活跃实例：

```bash
# 查看当前 Leader
kubectl get lease -n kube-system kube-controller-manager
```

| 协议 | 端口 | 说明 |
|------|------|------|
| **TCP** | **10257** | kube-controller-manager HTTPS 端口。提供健康检查（`/healthz`）和 Prometheus 指标（`/metrics`），默认仅允许 apiserver 访问 |

## 1.5 cloud-controller-manager

cloud-controller-manager 将云厂商相关的控制逻辑从 kube-controller-manager 中分离出来，使 K8s 核心与云厂商解耦。

### 包含的控制器

| 控制器 | 职责 |
|--------|------|
| **Node Controller** | 在云中创建/删除节点，更新节点信息 |
| **Route Controller** | 配置云网络路由（如 AWS VPC 路由表） |
| **Service Controller** | 创建/更新/删除 LoadBalancer 类型的 Service |
| **Volume Controller** | 创建/附加/扩展云存储卷 |

```bash
# 查看 cloud-controller-manager 状态
kubectl get pods -n kube-system -l component=cloud-controller-manager
```

### 端口

| 协议 | 端口 | 说明 |
|------|------|------|
| **TCP** | **10258** | cloud-controller-manager HTTPS 端口。提供健康检查和指标暴露，默认仅允许 apiserver 访问 |
