# 01 - 控制平面组件

控制平面（Control Plane）是 K8s 集群的"大脑"，负责维护集群状态、调度决策和 API 服务。传统上运行在 Master 节点，生产环境通常为高可用部署多副本。

## 整体架构

```
                    ┌─────────────────────────────────────┐
                    │          Control Plane (Master)      │
                    │                                     │
  kubectl ────────► │  kube-apiserver ──────► etcd         │
  API 请求          │       │                             │
                    │       ├────► kube-scheduler          │
                    │       │                             │
                    │       └────► kube-controller-manager │
                    │                                     │
                    │  cloud-controller-manager (可选)     │
                    └─────────────────────────────────────┘
```

所有组件都通过 kube-apiserver 交互，**apiserver 是唯一直接访问 etcd 的组件**。

---

## 1.1 kube-apiserver

kube-apiserver 是集群的统一入口，提供 RESTful API 供所有客户端（kubectl、kubelet、controller-manager、scheduler）通信。

### 核心职责

| 职责 | 说明 |
|------|------|
| **API 服务** | 提供 RESTful 接口，CRUD 集群资源 |
| **认证（Authentication）** | 验证请求者身份（X509 证书、Bearer Token、OIDC 等） |
| **鉴权（Authorization）** | 验证请求权限（RBAC、ABAC、Webhook） |
| **准入控制（Admission）** | 在写入前拦截和修改请求（Mutating → Validating） |
| **Watch 机制** | 支持长连接监听资源变更，驱动事件驱动架构 |

### 请求处理流程

```
请求 → 认证 → 鉴权 → Mutating Admission → Schema 校验 → Validating Admission → etcd
```

**三个阶段各有明确分工：**

1. **认证**：你是谁？（X509、Token、OIDC、ServiceAccount）
2. **鉴权**：你能做什么？（RBAC 最常用，Role/ClusterRole + Binding）
3. **准入控制**：请求是否合规？可在此阶段注入 sidecar、添加标签、校验策略

### 关键特性

- **API 聚合层**：通过 APIService 注册自定义 API Server，扩展原生 API
- **CRD（Custom Resource Definition）**：无需编写代码即可定义新资源类型
- **Watch 缓存**：对 etcd 的 watch 结果做本地缓存，减少 etcd 压力
- **API Priority and Fairness**：API 限流和优先级控制，防止过载

### 常用命令

```bash
# 查看 apiserver 的 API 版本
kubectl api-versions

# 查看所有可用资源
kubectl api-resources

# 查看 RBAC 授权
kubectl get clusterroles,clusterrolebindings
```

### 端口

| 协议 | 端口 | 说明 |
|------|------|------|
| **TCP** | **6443** | kube-apiserver HTTPS 服务端口，集群所有 API 请求的唯一入口。kubectl、kubelet、controller-manager、scheduler 均通过此端口与 apiserver 通信 |

> 防火墙/安全组必须放行此端口。生产环境通常在前面挂负载均衡器（如 HAProxy、AWS NLB）。

## 1.2 etcd

etcd 是 K8s 的状态存储后端，一个高可用的分布式键值数据库，使用 Raft 共识算法保证一致性。

### 核心特点

| 特性 | 说明 |
|------|------|
| **强一致性** | 所有读操作返回最新数据（线性一致性） |
| **Watch 支持** | 监听 key 或 key 前缀的变更事件 |
| **事务操作** | 支持条件写入（If-Then-Else） |
| **TTL** | 键可设置过期时间 |
| **Raft 共识** | 多数派写入，容忍少数节点故障 |

### 数据存储结构

etcd 中所有 K8s 资源存储在 `/registry` 前缀下：

```
/registry/pods/<namespace>/<pod-name>
/registry/deployments/<namespace>/<deployment-name>
/registry/services/specs/<namespace>/<service-name>
/registry/namespaces/<namespace>
/registry/nodes/<node-name>
/registry/configmaps/<namespace>/<name>
```

数据以 protobuf 序列化的格式存储，每个资源对象包含 `metadata.resourceVersion`，用于乐观锁和 Watch 机制。

### 关键概念

- **resourceVersion**：每个资源的版本号，Watch 基于此实现增量同步
- **compact revision**：etcd 定期压缩历史版本，过老的 Watch 会收到 `Gone` 错误需要重新 List
- **Quota**：默认 2GB 存储上限，生产建议 8GB

### 端口

| 协议 | 端口 | 说明 |
|------|------|------|
| **TCP** | **2379** | etcd 客户端 API 端口。kube-apiserver 通过此端口读写 etcd 数据，是集群的关键数据路径 |
| **TCP** | **2380** | etcd 集群内部 peer 通信端口。用于 leader/follower 之间的 Raft 复制和选举 |

> 2379 只需对 apiserver 开放；2380 只需在 etcd 节点间互通，不应暴露给外部。

### 运维要点

```bash
# 查看 etcd 集群状态（需要证书）
etcdctl --endpoints=https://127.0.0.1:2379 \
  --cacert=/etc/kubernetes/pki/etcd/ca.crt \
  --cert=/etc/kubernetes/pki/etcd/server.crt \
  --key=/etc/kubernetes/pki/etcd/server.key \
  endpoint status --write-out=table

# 备份
etcdctl snapshot save /backup/etcd-snapshot.db

# 恢复
etcdctl snapshot restore /backup/etcd-snapshot.db --data-dir=/var/lib/etcd-restore
```

## 1.3 kube-scheduler

kube-scheduler 负责将未绑定节点的 Pod 分配到最合适的 Node 上。

### 调度流程

```
Pod 进入调度队列
       │
       ▼
┌──────────────┐
│  Scheduling   │  1. PreFilter → 2. Filter → 3. PostFilter
│  Cycle        │  4. Score → 5. Reserve → 6. Bind
└──────────────┘
```

**两阶段决策：**

1. **Filter（过滤）**：排除不满足约束的节点
   - 资源不足（CPU/Memory/存储）
   - 节点选择器（nodeSelector）不匹配
   - 亲和性/反亲和性（affinity/anti-affinity）不满足
   - 污点容忍（Taint/Toleration）不匹配
   - 端口冲突

2. **Score（打分）**：对通过过滤的节点评分
   - 资源均衡度（LeastRequestedPriority）
   - 节点亲和性偏好
   - 数据本地性
   - Pod 间反亲和偏好
   - 自定义打分插件

### 影响调度的关键配置

```yaml
# nodeSelector - 简单的标签匹配
nodeSelector:
  disk: ssd

# nodeAffinity - 更灵活的节点亲和性
affinity:
  nodeAffinity:
    requiredDuringSchedulingIgnoredDuringExecution:   # 硬性约束
      nodeSelectorTerms:
      - matchExpressions:
        - key: topology.kubernetes.io/zone
          operator: In
          values: [zone-a, zone-b]
    preferredDuringSchedulingIgnoredDuringExecution:  # 软性偏好
    - weight: 80
      preference:
        matchExpressions:
        - key: node-type
          operator: In
          values: [high-performance]

# podAntiAffinity - Pod 间反亲和
  podAntiAffinity:
    requiredDuringSchedulingIgnoredDuringExecution:
    - labelSelector:
        matchExpressions:
        - key: app
          operator: In
          values: [redis]
      topologyKey: kubernetes.io/hostname

# Taints & Tolerations - 污点与容忍
tolerations:
- key: "dedicated"
  operator: "Equal"
  value: "gpu"
  effect: "NoSchedule"
```

### PriorityClass 与抢占

```yaml
apiVersion: scheduling.k8s.io/v1
kind: PriorityClass
metadata:
  name: high-priority
value: 1000000
globalDefault: false
preemptionPolicy: PreemptLowerPriority  # 允许抢占低优先级 Pod
```

高优先级 Pod 在资源不足时可以驱逐低优先级 Pod，腾出资源。

### 端口

| 协议 | 端口 | 说明 |
|------|------|------|
| **TCP** | **10259** | kube-scheduler HTTPS 端口。提供健康检查（`/healthz`）和 Prometheus 指标（`/metrics`），默认仅允许 apiserver 访问 |

## 1.4 kube-controller-manager

kube-controller-manager 是多个控制器的集合体，每个控制器运行独立的 **控制循环（Reconciliation Loop）**，持续将系统实际状态推向期望状态。

### 核心控制循环原理

```
        ┌──────────────────────────────────┐
        │         Controller Loop          │
        │                                  │
        │  观察当前状态 → 计算差异 → 执行动作  │
        │       ▲                    │     │
        │       └────────────────────┘     │
        │         (持续循环)                │
        └──────────────────────────────────┘
```

这个模式被称为 **"水平触发"**（level-triggered）而非"边缘触发"：控制器不关心事件本身，只关心当前状态与期望状态的差异。即使中间丢失了某些事件，最终也能收敛到正确状态。

### 主要控制器

| 控制器 | 职责 |
|--------|------|
| **Node Controller** | 监控 Node 状态，处理 Node 不可达时的 Pod 驱逐 |
| **ReplicaSet Controller** | 确保 RS 管理的 Pod 副本数与期望一致 |
| **Deployment Controller** | 管理 ReplicaSet，实现滚动更新和回滚 |
| **Endpoints Controller** | 维护 Service 与 Pod 的映射关系（Endpoints 对象） |
| **ServiceAccount Controller** | 为每个 Namespace 创建默认 ServiceAccount |
| **Namespace Controller** | 处理 Namespace 删除时的级联清理 |
| **PersistentVolume Controller** | 管理 PV 与 PVC 的绑定 |
| **Job Controller** | 管理 Job 的执行和完成状态 |
| **CronJob Controller** | 按 Cron 表达式创建 Job |

### Leader Election

kube-controller-manager 多副本部署时，通过 etcd 的 Leader Election 机制保证同一时刻只有一个活跃实例：

```bash
# 查看当前 Leader
kubectl get lease -n kube-system kube-controller-manager
```

### 端口

| 协议 | 端口 | 说明 |
|------|------|------|
| **TCP** | **10257** | kube-controller-manager HTTPS 端口。提供健康检查（`/healthz`）和 Prometheus 指标（`/metrics`），默认仅允许 apiserver 访问 |

## 1.5 cloud-controller-manager

cloud-controller-manager 将云厂商相关的控制逻辑从 kube-controller-manager 中分离出来，使 K8s 核心与云厂商解耦。

### 包含的控制器

| 控制器 | 职责 |
|--------|------|
| **Node Controller** | 在云中创建/删除节点，更新节点信息 |
| **Route Controller** | 配置云网络路由（如 AWS VPC 路由表） |
| **Service Controller** | 创建/更新/删除 LoadBalancer 类型的 Service |
| **Volume Controller** | 创建/附加/扩展云存储卷 |

```bash
# 查看 cloud-controller-manager 状态
kubectl get pods -n kube-system -l component=cloud-controller-manager
```

### 端口

| 协议 | 端口 | 说明 |
|------|------|------|
| **TCP** | **10258** | cloud-controller-manager HTTPS 端口。提供健康检查和指标暴露，默认仅允许 apiserver 访问 |

## 本章小结

| 组件 | 一句话总结 | 端口 | 关键概念 |
|------|-----------|------|---------|
| kube-apiserver | 集群唯一入口，认证→鉴权→准入控制 | TCP 6443 | RBAC, Admission, Watch |
| etcd | 状态存储后端，Raft 一致性 | TCP 2379/2380 | resourceVersion, 快照备份 |

| controller-manager | 多个控制循环，水平触发收敛 | TCP 10257 | Reconciliation Loop, Leader Election |
| cloud-controller-manager | 云厂商集成 | TCP 10258 | Node/Route/Service/Volume 控制器 |
