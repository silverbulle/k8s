# 10 - 案例：弹性伸缩

本案例演示如何构建完整的弹性伸缩体系，覆盖 Pod 水平伸缩（HPA）、Pod 垂直伸缩（VPA）和集群节点伸缩（Cluster Autoscaler）三个层次。

## 伸缩体系总览

```
流量变化
    │
    ├── 短期波动（秒~分钟）
    │     └─► HPA：自动增减 Pod 副本数
    │           依赖：Metrics Server / Prometheus
    │
    ├── 资源利用率优化（小时~天）
    │     └─► VPA：自动调整 Pod 的 requests/limits
    │           依赖：VPA Recommender
    │
    └── 长期容量规划（天~周）
          └─► Cluster Autoscaler：自动增减 Node 数量
                依赖：云厂商 API / 节点组
```

---

## 场景设定

部署一个电商 Web 应用，流量特征：

- **日常基线**：100 QPS，3 个 Pod 即可承载
- **促销活动**：2000 QPS，需要 15-20 个 Pod
- **活动结束**：流量骤降，需要快速缩容
- **凌晨低谷**：10 QPS，2 个 Pod 即可

## 第二步：HPA — 水平 Pod 伸缩

### 基于 CPU 的 HPA

```yaml
# 02-hpa-cpu.yaml
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: webapp-hpa-cpu
  namespace: webapp
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: webapp
  minReplicas: 2
  maxReplicas: 20
  metrics:
  - type: Resource
    resource:
      name: cpu
      target:
        type: Utilization
        averageUtilization: 65       # 目标 CPU 利用率 65%
```

**伸缩计算：**

```
当前: 3 Pod, 平均 CPU 80%
目标: 65%
期望副本数 = ceil[3 × (80/65)] = ceil[3.69] = 4
→ 扩容到 4 个 Pod

当前: 4 Pod, 平均 CPU 30%（活动结束）
目标: 65%
期望副本数 = ceil[4 × (30/65)] = ceil[1.85] = 2
→ 缩容到 2 个 Pod（受 minReplicas=2 限制）
```

### 基于内存的 HPA

```yaml
# 03-hpa-memory.yaml
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: webapp-hpa-memory
  namespace: webapp
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: webapp
  minReplicas: 2
  maxReplicas: 20
  metrics:
  - type: Resource
    resource:
      name: memory
      target:
        type: Utilization
        averageUtilization: 75       # 内存使用率 > 75% 扩容
```

### 基于自定义指标的 HPA（QPS）

需要 Prometheus Adapter 将 Prometheus 指标转换为 K8s custom metrics。

```yaml
# 04-hpa-custom.yaml
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: webapp-hpa-qps
  namespace: webapp
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: webapp
  minReplicas: 3
  maxReplicas: 30
  metrics:
  # CPU 指标（保底）
  - type: Resource
    resource:
      name: cpu
      target:
        type: Utilization
        averageUtilization: 70
  # 自定义指标：每秒请求数
  - type: Pods
    pods:
      metric:
        name: http_requests_per_second
      target:
        type: AverageValue
        averageValue: "100"          # 每个 Pod 平均 100 QPS 时扩容

  # 伸缩行为精细控制
  behavior:
    scaleUp:
      stabilizationWindowSeconds: 0   # 扩容立即执行（不等待）
      policies:
      - type: Pods
        value: 4                      # 每次最多扩 4 个
        periodSeconds: 60
      - type: Percent
        value: 50                     # 或每次最多增加当前数量的 50%
        periodSeconds: 60
      selectPolicy: Max               # 取两个策略中扩容更多的

    scaleDown:
      stabilizationWindowSeconds: 300  # 缩容前等待 5 分钟（防抖动）
      policies:
      - type: Pods
        value: 2                       # 每次最多缩 2 个
        periodSeconds: 120             # 每 2 分钟执行一次
```

### Prometheus Adapter 配置

```yaml
# 05-prometheus-adapter-config.yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: adapter-config
  namespace: monitoring
data:
  config.yaml: |
    rules:
    # 将 Prometheus 的 http_requests_total 转换为 QPS 指标
    - seriesQuery: 'http_requests_total{namespace!="",pod!=""}'
      resources:
        overrides:
          namespace: {resource: "namespace"}
          pod: {resource: "pod"}
      name:
        matches: "^(.*)_total"
        as: "${1}_per_second"
      metricsQuery: |
        sum(rate(<<.Series>>{<<.LabelMatchers>>}[2m])) by (<<.GroupBy>>)
```

## 第三步：VPA — 垂直 Pod 伸缩

VPA 自动调整 Pod 的 requests/limits，避免资源浪费或不足。

```yaml
# 06-vpa.yaml
apiVersion: autoscaling.k8s.io/v1
kind: VerticalPodAutoscaler
metadata:
  name: webapp-vpa
  namespace: webapp
spec:
  targetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: webapp
  updatePolicy:
    updateMode: "Auto"               # 自动应用推荐（会重启 Pod）
  resourcePolicy:
    containerPolicies:
    - containerName: "*"
      minAllowed:
        cpu: "50m"
        memory: "64Mi"
      maxAllowed:
        cpu: "2"
        memory: "2Gi"
      controlledResources: ["cpu", "memory"]
```

### VPA 推荐查看

```bash
# 查看 VPA 推荐值
kubectl get vpa -n webapp
NAME         MODE   CPU    MEM     PROVIDED   AGE
webapp-vpa   Auto   150m   192Mi   True       7d

# 查看详细推荐
kubectl get vpa webapp-vpa -n webapp -o yaml
status:
  recommendation:
    containerRecommendations:
    - containerName: webapp
      lowerBound:          # 最低推荐值
        cpu: "100m"
        memory: "128Mi"
      target:              # 推荐值
        cpu: "150m"
        memory: "192Mi"
      upperBound:          # 最高推荐值
        cpu: "200m"
        memory: "256Mi"
      uncappedTarget:      # 不考虑 min/max 限制的推荐值
        cpu: "80m"
        memory: "96Mi"
```

### VPA 模式对比

| 模式 | 行为 | 适用场景 |
|------|------|---------|
| `Off` | 只推荐，不修改 Pod | 先观察再决定是否启用 |
| `Initial` | 仅在 Pod 创建时设置资源 | 避免运行中重启 |
| `Auto` | 自动调整并重启 Pod | 信任 VPA 推荐 |

## 第四步：Cluster Autoscaler — 集群节点伸缩

### 云环境部署（以 AWS EKS 为例）

```yaml
# 07-cluster-autoscaler.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: cluster-autoscaler
  namespace: kube-system
spec:
  replicas: 1
  selector:
    matchLabels:
      app: cluster-autoscaler
  template:
    metadata:
      labels:
        app: cluster-autoscaler
    spec:
      serviceAccountName: cluster-autoscaler
      containers:
      - name: cluster-autoscaler
        image: registry.k8s.io/autoscaling/cluster-autoscaler:v1.29.0
        command:
        - ./cluster-autoscaler
        - --v=4
        - --stderrthreshold=info
        - --cloud-provider=aws
        - --skip-nodes-with-local-storage=false
        - --expander=least-waste        # 选择浪费最少的节点组
        - --node-group-auto-discovery=asg:tag=k8s.io/cluster-autoscaler/enabled
        - --balance-similar-node-groups
        - --scale-down-delay-after-add=10m    # 扩容后 10 分钟内不缩容
        - --scale-down-unneeded-time=10m      # 空闲 10 分钟才缩容
        - --scale-down-utilization-threshold=0.5  # 利用率 < 50% 视为空闲
        - --max-node-provision-time=15m
        resources:
          requests:
            cpu: 100m
            memory: 300Mi
```

### 伸缩触发逻辑

```
扩容触发：
  Pod 处于 Pending 状态
  → scheduler 无法找到满足条件的 Node
  → Cluster Autoscaler 检查节点组
  → 向云厂商请求新增 Node（AWS ASG / GCP MIG）
  → 新 Node 启动并加入集群（通常 2-5 分钟）
  → Pending Pod 调度成功

缩容触发：
  Node 上所有 Pod 可以迁移到其他 Node
  → Node 利用率 < 50% 持续 10 分钟
  → Cluster Autoscaler 标记为可缩容
  → 驱逐该 Node 上的 Pod（尊重 PDB）
  → 等待 Pod 迁移完成
  → 从云厂商删除 Node
```

## 第五步：Pod Disruption Budget（PDB）

PDB 保护关键应用在缩容/维护期间保持最低可用副本数。

```yaml
# 08-pdb.yaml
apiVersion: policy/v1
kind: PodDisruptionBudget
metadata:
  name: webapp-pdb
  namespace: webapp
spec:
  # 两种方式（互斥）：
  # minAvailable: 2                  # 最少保持 2 个可用
  maxUnavailable: 1                  # 最多允许 1 个不可用
  selector:
    matchLabels:
      app: webapp
```

**PDB 生效场景：**

| 操作 | 是否受 PDB 保护 |
|------|----------------|
| `kubectl drain`（节点维护） | ✓ 受保护 |
| Cluster Autoscaler 缩容 | ✓ 受保护 |
| 节点故障（kubelet 无响应） | ✗ 不受保护 |
| 用户直接 `kubectl delete pod` | ✗ 不受保护 |

## 完整弹性方案验证

### 模拟流量高峰

```bash
# 1. 查看初始状态
kubectl get pods,hpa -n webapp
NAME                        READY   REPLICAS
deployment/webapp           3/3     3
NAME                        REFERENCE          TARGETS   MINPODS   MAXPODS   REPLICAS
hpa/webapp-hpa-qps          Deployment/webapp   45/100    3         30        3

# 2. 发起压力测试
kubectl run load-generator --image=busybox -n webapp --rm -it -- \
  sh -c "while true; do wget -q -O- http://webapp-svc/api/products; done"

# 3. 观察 HPA 响应（另一个终端）
kubectl get hpa -n webapp -w
# 30s 后 CPU 指标上升
webapp-hpa-qps    85/100    3    30    3
webapp-hpa-qps    120/100   3    30    3→5
webapp-hpa-qps    150/100   3    30    5→8
webapp-hpa-qps    95/100    3    30    8→12

# 4. 如果 Pod 扩到当前 Node 资源不足
kubectl get nodes
# 新 Node 自动加入
node-3   Ready    <none>   2m    ← Cluster Autoscaler 扩容

# 5. 停止压力测试后
kubectl get hpa -n webapp -w
# 等待 5 分钟稳定窗口后开始缩容
webapp-hpa-qps    30/100    3    30    12→10→8→5→3
```

### 全链路伸缩时序

```
时间    事件
───────────────────────────────────────────────
T+0s    流量开始上升
T+30s   HPA 检测到 CPU/QPS 超阈值
T+30s   HPA 扩容 Pod（3→5→8→12）
T+5m    Pod 数量到达上限，Node 资源不足
T+5m    Cluster Autoscaler 触发 Node 扩容
T+8m    新 Node Ready，Pending Pod 调度成功
T+8m    系统稳定承载高流量

T+30m   流量回落
T+35m   HPA 稳定窗口结束，开始缩容 Pod
T+45m   Pod 缩至 minReplicas
T+55m   Node 利用率下降
T+65m   Cluster Autoscaler 缩容 Node
T+65m   系统恢复基线
```

## 生产环境最佳实践

### 1. 合理设置 requests/limits

```yaml
resources:
  requests:
    cpu: "100m"         # 基于 VPA 推荐值
    memory: "128Mi"
  limits:
    cpu: "500m"         # CPU limit 建议 3-5x request
    memory: "256Mi"     # Memory limit 建议 1.5-2x request
```

### 2. HPA + VPA 配合

```yaml
# HPA 负责副本数（应对流量波动）
# VPA 负责单 Pod 资源（优化利用率）
# 两者不要同时管理同一维度（会导致冲突）
# 推荐：HPA 管 CPU 副本数，VPA 管 Memory 资源配置
```

### 3. 预热与冷启动

```yaml
spec:
  template:
    spec:
      containers:
      - name: webapp
        # Startup Probe 给慢启动应用充足时间
        startupProbe:
          httpGet:
            path: /health
            port: 8080
          failureThreshold: 60
          periodSeconds: 5         # 最多等 5 分钟
        # Readiness Probe 确保只有就绪的 Pod 接收流量
        readinessProbe:
          httpGet:
            path: /ready
            port: 8080
          periodSeconds: 5
```

### 4. 缩容保护

```yaml
# PDB 保证最低可用
maxUnavailable: 1

# HPA 缩容稳定窗口
behavior:
  scaleDown:
    stabilizationWindowSeconds: 300  # 5 分钟稳定窗口

# 优雅终止
terminationGracePeriodSeconds: 60
lifecycle:
  preStop:
    exec:
      command: ["/bin/sh", "-c", "sleep 15"]
```

## 知识点回顾

| 组件 | 伸缩维度 | 触发条件 | 典型延迟 |
|------|---------|---------|---------|
| **HPA** | Pod 副本数 | CPU/内存/自定义指标超阈值 | 30s-2min |
| **VPA** | Pod 资源量 | 实际使用与 request 偏差大 | 重启 Pod |
| **Cluster Autoscaler** | Node 数量 | Pending Pod / 低利用率 Node | 2-10min |
| **PDB** | 保护最小可用 | 缩容/维护操作 | 即时 |
