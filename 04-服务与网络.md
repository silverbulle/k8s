# 04 - 服务与网络

K8s 的网络模型解决了三个层次的问题：**Pod 之间如何通信**（CNI 插件实现）、**集群内服务发现**（Service + DNS）、**外部如何访问集群**（Ingress）。本章从底层网络到上层服务逐层展开。

## K8s 网络基础

K8s 的网络设计遵循以下原则：

1. **每个 Pod 拥有独立 IP**（不需要 NAT）
2. **所有 Pod 之间可以直接通信**（扁平网络，无防火墙限制）
3. **Node 可以与该 Node 上的所有 Pod 通信**

```
┌──────────────────────────────────────────────┐
│              K8s Cluster                      │
│                                              │
│  Node-1                   Node-2             │
│  ┌─────────┐ ┌─────────┐ ┌─────────┐        │
│  │ Pod A   │ │ Pod B   │ │ Pod C   │        │
│  │10.244   │ │10.244   │ │10.244   │        │
│  │ .1.5    │◄┤ .1.8    │◄┤ .2.3    │        │
│  └─────────┘ └─────────┘ └─────────┘        │
│                                              │
│  Service: 10.96.0.10 (虚拟 IP，通过 iptables 转发)
└──────────────────────────────────────────────┘
```

**Pod IP 是临时的**——Pod 重建后 IP 会变。Service 提供了稳定的访问入口。

## 4.1 CNI 与 Pod 网络

K8s 自身不实现 Pod 网络，而是通过 **CNI（Container Network Interface）** 规范将网络实现委托给第三方插件。CNI 插件负责：Pod IP 分配、跨节点 Pod 通信、NetworkPolicy 执行。

### CNI 工作流程

```
kubelet 创建 Pod
  │
  ├─ 1. 调用 CRI 创建 Pod Sandbox（Pause 容器）
  │
  ├─ 2. kubelet 调用 CNI 插件
  │     ┌───────────────────────────────────────────┐
  │     │  CNI ADD 请求：                           │
  │     │  - Container ID                          │
  │     │  - Network Namespace (Pause 容器的 ns)    │
  │     │  - 接口名: eth0                           │
  │     └───────────────────────────────────────────┘
  │
  ├─ 3. CNI 插件执行：
  │     - 分配 IP 地址（IPAM）
  │     - 配置网络接口（veth pair）
  │     - 设置路由规则
  │     - 将 Pod 网络接入集群网络
  │
  └─ 4. 返回结果给 kubelet：
        { "ip4": { "ip": "10.244.1.5/24", "gateway": "10.244.1.1" } }
```

### CNI 配置文件

kubelet 启动时读取 CNI 配置（通常在 `/etc/cni/net.d/`）：

```json
// /etc/cni/net.d/10-calico.conflist
{
  "name": "k8s-pod-network",
  "cniVersion": "0.3.1",
  "plugins": [
    {
      "type": "calico",
      "log_level": "info",
      "datastore_type": "kubernetes",
      "mtu": 1440,
      "ipam": {
        "type": "calico-ipam"
      },
      "policy": {
        "type": "k8s"
      },
      "kubernetes": {
        "kubeconfig": "/etc/cni/net.d/calico-kubeconfig"
      }
    },
    {
      "type": "portmap",
      "snat": true,
      "capabilities": {"portMappings": true}
    },
    {
      "type": "bandwidth",
      "capabilities": {"bandwidth": true}
    }
  ]
}
```

### veth pair — Pod 网络的基石

每个 Pod 通过一对虚拟以太网设备（veth pair）连接到宿主机网络：

```
Node-1
┌──────────────────────────────────────────────────┐
│                                                  │
│  ┌────────────────┐        ┌────────────────┐   │
│  │ Pod A          │        │ Pod B          │   │
│  │ ┌────────────┐ │        │ ┌────────────┐ │   │
│  │ │  eth0      │ │        │ │  eth0      │ │   │

│  └────────┼────────┘        └────────┼────────┘   │

│  ┌────────┴─────┐        ┌────────┴─────┐       │

│           └────────────┬──────────┘              │
│                        ▼                         │
│           ┌──────────────────────┐               │
│           │   宿主机路由表        │               │
```

## Calico 架构

Calico 是生产环境最常用的 CNI 插件之一，由 Project Calico 开发，Tigera 维护。它的一个关键设计是**数据平面的跨节点通信方式可切换**——BGP 纯三层路由、VXLAN 封装、IPIP 封装，三种模式下**参与的组件并不相同**。因此理解 Calico 架构要分两层：先看**模式无关**的基础组件，再看**模式相关**的数据平面协同。

> 重要前提：**BIRD 这个组件是 BGP 模式专用的，VXLAN/IPIP 模式下根本不运行。** 把 BIRD 当成 Calico 的"标配核心组件"是常见误解——下面会分开讲清楚。

### 模式无关：基础组件总览

无论哪种网络模式，以下组件都在运行，职责不变：

```
┌────────────────────────────────────────────────────────────┐
│              Calico 基础组件（所有模式通用）                  │
│                                                            │
│  每个 Node 上运行（DaemonSet = calico-node Pod）:            │
│  ┌──────────────────────────────────────────────────┐      │
│  │  Felix         — 数据平面核心：编程路由/iptables/  │      │
│  │                   邻居表/隧道设备，执行 NetworkPolicy│     │
│  │  (BIRD/confd)  — 仅 BGP 模式运行，见下文           │      │
│  └──────────────────────────────────────────────────┘      │
│                                                            │
│  集群级运行（Deployment）:                                    │
│  ┌──────────────────────────────────────────────────┐      │
│  │  calico-kube-controllers                          │      │
│  │  ├── Node Controller      — Node 生命周期管理      │      │
│  │  ├── Policy Controller    — NetworkPolicy 同步     │      │
│  │  ├── IPAM Controller      — IP 地址管理            │      │
│  │  └── ServiceAccount Ctrl  — SA Token 管理         │      │
│  └──────────────────────────────────────────────────┘      │
│                                                            │
│  可选组件（大规模集群）:                                      │
│  ┌──────────────────────────────────────────────────┐      │
│  │  Typha           — Felix 与 datastore 的代理       │      │
│  │  calico-typha    — 大规模集群减少 apiserver 压力   │      │
│  └──────────────────────────────────────────────────┘      │
└────────────────────────────────────────────────────────────┘
```

**为什么 BIRD/confd 标成"仅 BGP 模式"？** 因为跨节点路由的学习方式不同：
- **BGP 模式**：用 BGP 协议在节点间动态交换路由 → 需要 BIRD 跑协议、confd 生成配置
- **VXLAN/IPIP 模式**：Felix 直接从 datastore 读远端节点 VTEP 信息，**自己编程内核**（路由表 + FDB + 邻居表 + 隧道设备），不需要任何协议守护进程

### Felix — 数据平面核心（所有模式都在）

Felix 运行在每个 Node 上，是 Calico 的核心数据平面组件。**两种模式下它都在跑，但编进内核的东西不同**：

| 职责 | BGP 模式 | VXLAN/IPIP 模式 |
|------|:--------:|:---------------:|
| NetworkPolicy（iptables/nftables 规则） | ✅ | ✅ |
| veth pair 创建配置 | ✅ | ✅ |
| 同节点 Pod 的 ARP/NDP 代理 | ✅ | ✅ |
| 跨节点路由写入宿主机路由表 | ✅ | ✅ |
| 跨节点路由的**来源** | BIRD 通过 BGP 学到后写 | Felix 从 datastore 读 VTEP 直接写 |
| 隧道设备（vxlan.calico / tunl0）创建与 FDB/邻居表编程 | ❌ | ✅ |

```
Felix 的统一数据流（所有模式）：

  Datastore (etcd/K8s API)
       │
       │ Watch 资源变更（Pod、WorkloadEndpoint、IPAM Block、NetworkPolicy）
       ▼
  ┌─────────┐
  │  Felix  │ ──► 写 iptables/nftables（NetworkPolicy）
  └─────────┘ ──► 管理 veth pair、响应同节点 ARP
            ──► 写路由表（跨节点路由，来源视模式而定）
            ──► [仅封装模式] 编程隧道设备 + FDB + 邻居表
```

### calico-kube-controllers

calico-kube-controllers 是集群级的控制平面组件，包含多个子控制器：

| 控制器 | 职责 |
|--------|------|
| **Node Controller** | 监听 K8s Node 事件，同步创建/删除 Calico Node 资源；Node 删除时清理该 Node 上的 IPAM 分配 |
| **Policy Controller** | 监听 K8s NetworkPolicy，转换为 Calico NetworkPolicy 格式供 Felix 消费 |
| **IPAM Controller** | 管理 IP 地址分配，处理 Node 删除时的 IP 回收 |
| **ServiceAccount Controller** | 同步 K8s ServiceAccount 到 Calico 策略 |

### Calico IPAM — IP 地址管理（基于 Block）

```
IP 分配流程：

1. kubelet 创建 Pod → 调用 CNI
2. Calico CNI 请求 IPAM 分配 IP
3. IPAM 从 IPPool 中分配，以 block 为单位亲和到 Node：
   - 每个 block 默认 /26（64 个 IP），block 与 Node 绑定（有节点亲和性）
   - 优先从当前 Node 已持有的 block 中分配
   - 本节点 block 用尽时申请新 block（可能跨节点借用）
4. IP 与 Pod 绑定记录到 datastore（K8s CRD 或 etcd）
```

> **Block 是理解 Calico 路由的关键**：跨节点路由是**按 block（/26）而不是按 Node CIDR（/24）**下发的。一个 Node 可能持有多个 block，Felix 通过 datastore 里的 block→node 映射知道每个 block 归谁。下面 VXLAN 协同部分会用到这一点。

```yaml
apiVersion: projectcalico.org/v3
kind: IPPool
metadata:
  name: default-ipv4-pool
spec:
  cidr: 10.244.0.0/16            # Pod 地址池
  blockSize: 26                   # 每个 block /26（64 个 IP）
  ipipMode: Never                 # 不使用 IPIP 封装
  vxlanMode: Never                # 不使用 VXLAN 封装 → 纯 BGP 模式
  natOutgoing: true               # Pod 访问外部时做 SNAT
  nodeSelector: all()
```

## Calico 网络模式与组件协同

三种模式最大的区别就是**"跨节点路由怎么学、谁编进内核"**。下面分开讲每种模式下组件如何协同。

### 模式一：BGP 纯三层路由（组件：Felix + BIRD + confd）

BGP 模式下，跨节点路由通过 **BGP 协议**动态交换，BIRD 是数据平面主角之一：

```
协同链路：

  datastore ──Watch──► confd ──生成──► BIRD 配置
                                          │
                                          ▼
  Node-1  ┌────────────┐   BGP Session   ┌────────────┐  Node-2
          │  BIRD      │◄────TCP:179────►│  BIRD      │
          │            │                 │            │
          │ 宣告:       │                 │ 宣告:       │
          │ 10.244.1.0 │                 │ 10.244.2.0 │
          │  /26       │                 │  /26       │
          └─────┬──────┘                 └─────┬──────┘
                │ 学到的路由                     │ 学到的路由
                ▼                              ▼
          ┌──────────┐                   ┌──────────┐
          │  Felix   │ 写入宿主机路由表     │  Felix   │
          └──────────┘                   └──────────┘
```

**三方分工：**
- **confd**：watch datastore，把本节点的 block/节点信息渲染成 BIRD 配置文件
- **BIRD**：跑 BGP 协议，与其它节点的 BIRD 建立 session（TCP 179），互相宣告自己持有的 block 路由，并学习远端 block 路由
- **Felix**：把 BIRD 学到的远端路由写入宿主机路由表（`ip route`），并负责 NetworkPolicy、veth 等通用职责

**路由结果（Node-1 的路由表）：**
```
10.244.2.0/26 via 192.168.1.11    ← BIRD 通过 BGP 学到，Felix 写入
10.244.1.0/26 dev calixxx         ← 本地直连 block
```

**BGP 拓扑模式：**

| 模式 | 说明 | 适用场景 |
|------|------|---------|
| **Full Mesh** | 每个 Node 与其他所有 Node 建立 BGP Session | 中小集群（< 100 Node） |
| **Route Reflector** | 指定 RR 节点集中分发路由，减少 BGP Session 数 | 大规模集群 |
| **AS Per Rack** | 与物理路由器 BGP Peering，融入数据中心网络 | 物理网络集成 |

```yaml
# 配置 BGP Peer（Route Reflector 模式）
apiVersion: projectcalico.org/v3
kind: BGPPeer
metadata:
  name: route-reflector
spec:
  peerIP: 192.168.1.1
  asNumber: 64512
  nodeSelector: all()
```

**特点：** 性能最好（原始 IP 包无封装，tcpdump 直接可抓）；要求底层网络支持路由传播（二层可达或能与物理设备 BGP Peering）。适用于裸金属、VM 同子网、云环境 VPC 内。

### 模式二：VXLAN 封装（组件：仅 Felix，无 BIRD）

> 这是你问的核心。**VXLAN 模式下没有 BIRD、没有 confd，Felix 一肩挑**——它直接从 datastore 读远端节点的 VTEP 信息，自己把路由表、FDB、邻居表、隧道设备全部编进内核。

**协同链路（Felix 独自完成）：**

```
① 节点上线：每个 Node 的 Felix 把自己的 VTEP 信息
   （tunnel IP + tunnel MAC）写入 datastore

② 其它 Node 的 Felix 从 datastore 读到所有远端 Node 的 VTEP 信息，
   在本地内核编程三类状态：

   a. 路由表（按 block /26，不是按节点 CIDR /24）
      10.244.2.0/26 via <Node-2 tunnelIP> dev vxlan.calico onlink

   b. FDB 表（VXLAN 转发表：某 MAC → 某远端 Node IP）
      <Node-2 tunnelMAC> dev vxlan.calico dst <Node-2 tunnelIP>

   c. 邻居/ARP 表（预填，抑制 ARP 广播）
      10.244.2.x dev vxlan.calico lladdr <Node-2 tunnelMAC>

③ Felix 还创建 vxlan.calico 内核隧道设备（VNI=4096，UDP 4789）
```

**为什么没有 BIRD？** 因为路由不是"学"来的，是 Felix **直接从 datastore 拉取**的。datastore 里就有每个 block 归属哪个 Node、每个 Node 的 VTEP 是什么，Felix watch 到变化就直接写内核，不需要任何节点间协议。这比 BGP 更简单，代价是封装开销。

**一次跨节点 Pod 通信的完整数据流：**

```
Pod A (10.244.1.5@Node1) → Pod C (10.244.2.3@Node2)

1. Pod A 发包: src=10.244.1.5  dst=10.244.2.3
2. 经 veth 进入 Node1 宿主机网络栈
3. 查路由表（Felix 写的）: 10.244.2.0/26 via <Node2-tunnelIP> dev vxlan.calico
4. 内核把包交给 vxlan.calico 设备
5. vxlan.calico 查 FDB（Felix 写的）: <Node2-tunnelMAC> → Node2 物理IP
6. 内核封装：
     外层: src=Node1-IP, dst=Node2-IP, UDP dport=4789
     内层: 原始包 (src=10.244.1.5, dst=10.244.2.3)
7. 封装后的 UDP 包走底层物理网络送到 Node2
8. Node2 收到 UDP:4789 → 内核解封装 → 还原内层包
9. 内层包按 Pod IP 路由到 Pod C 的 veth → Pod C
```

**特点：** 不依赖底层路由，跨子网/跨 AZ 可用；缺点是封装开销（约 5-10% 性能损失），MTU 需减小 50 字节。

### 模式三：IPIP 封装（组件：仅 Felix，无 BIRD）

机制与 VXLAN 相同（Felix 直接编程内核，无 BIRD），区别只在封装协议——用 IP-in-IP（协议号 4）替代 VXLAN，隧道设备是 `tunl0` 而非 `vxlan.calico`：
- 开销比 VXLAN 小（20 字节 vs 50 字节）
- 部分云环境不支持（如 AWS 丢弃 IPIP 包）
- 逐渐被 VXLAN 取代

### 模式切换

```yaml
# 修改 IPPool 即可切换（Felix 会重新编程内核）
spec:
  ipipMode: Never        # Never | Always | CrossSubnet
  vxlanMode: Always      # Never | Always | CrossSubnet

# CrossSubnet = 同子网用 BGP，跨子网用封装（混合最佳实践）
```

> **三种模式组件对照速查：**
>
> | 模式 | 跨节点路由来源 | BIRD/confd | 隧道设备 | 封装端口 |
> |------|--------------|:----------:|---------|---------|
> | BGP | BGP 协议动态学习 | ✅ | 无 | 无 |
> | VXLAN | Felix 从 datastore 直接拉取 | ❌ | vxlan.calico | UDP 4789 |
> | IPIP | Felix 从 datastore 直接拉取 | ❌ | tunl0 | 协议号 4 |

## Calico NetworkPolicy 增强

Calico 扩展了 K8s 原生 NetworkPolicy，提供更多能力：

```yaml
# Calico 特有的策略能力
apiVersion: projectcalico.org/v3
kind: NetworkPolicy
metadata:
  name: advanced-policy
  namespace: production
spec:
  selector: app == 'web'              # 表达式选择器（比 matchLabels 更灵活）
  types:
  - Ingress
  - Egress
  ingress:
  - action: Allow
    source:
      selector: role == 'frontend'
      namespaceSelector: name == 'web'  # 跨 namespace 选择
    destination:
      ports:
      - 80
      - 443
  egress:
  - action: Allow
    destination:
      domains:                         # 域名级出站控制（K8s 原生不支持）
      - '*.example.com'
      - 'api.github.com'
      ports:
      - 443
  - action: Deny                        # 显式拒绝（K8s 原生只有 Allow）
    destination:
      nets:
      - 169.254.169.254/32             # 禁止访问云元数据服务
```

**Calico vs K8s NetworkPolicy 对比：**

| 特性 | K8s NetworkPolicy | Calico NetworkPolicy |
|------|-------------------|---------------------|
| 动作 | 只有 Allow | Allow / Deny / Log |
| 选择器 | matchLabels | 表达式（==, !=, in, not in） |
| 跨 Namespace | namespaceSelector | namespaceSelector + 表达式 |
| 域名过滤 | 不支持 | 支持 `domains` |
| 全局策略 | 不支持 | GlobalNetworkPolicy |
| 优先级 | 无（全部 Allow） | order 字段排序 |
| 日志 | 不支持 | action: Log |

## 主流 CNI 方案对比

| 特性 | Calico | Flannel | Cilium | Weave |
|------|--------|---------|--------|-------|
| **网络模式** | BGP / VXLAN / IPIP | VXLAN | eBPF / VXLAN | Weave Mesh |
| **数据平面** | iptables / BPF | VXLAN | eBPF（XDP/TC） | iptables |
| **NetworkPolicy** | 支持（增强） | 不支持 | 支持（增强） | 支持 |
| **性能** | 高（BGP 无封装） | 中（VXLAN 开销） | 极高（eBPF） | 中 |
| **可观测性** | 一般 | 基本 | 强（Hubble） | 一般 |
| **大规模集群** | 支持（Typha + RR） | 不推荐 | 推荐 | 不推荐 |
| **加密** | WireGuard | 不支持 | IPsec / WireGuard | 支持 |
| **维护方** | Tigera | CoreOS（已停更） | Isovalent | Weaveworks |

**选择建议：**
- **通用生产**：Calico（成熟稳定，社区支持好）
- **高性能/可观测**：Cilium（eBPF 数据平面，Hubble 可观测性）
- **简单测试**：Flannel（配置最简单，功能最少）

## 4.2 Service

Service 是一组 Pod 的逻辑抽象，提供稳定的网络端点（IP + 端口），后端 Pod 变化时自动更新。

### Service 类型

| 类型 | 访问范围 | 实现方式 |
|------|---------|---------|
| **ClusterIP**（默认） | 集群内部 | 分配虚拟 IP，通过 iptables/IPVS 转发 |
| **NodePort** | 集群外部 | 在每个 Node 上开放固定端口（30000-32767） |
| **LoadBalancer** | 集群外部 | 云厂商负载均衡器 |
| **ExternalName** | 外部服务 | CNAME 映射到外部域名 |

### ClusterIP

```yaml
apiVersion: v1
kind: Service
metadata:
  name: webapp-svc
spec:
  type: ClusterIP
  selector:
    app: webapp              # 匹配带有 app=webapp 标签的 Pod
  ports:
  - port: 80                 # Service 端口
    targetPort: 8080          # Pod 实际端口
    protocol: TCP
```

```
集群内访问方式：
  curl http://webapp-svc.default.svc.cluster.local:80
  curl http://webapp-svc:80              # 同 namespace 可缩写
```

**Headless Service**（ClusterIP: None）：

```yaml
spec:
  clusterIP: None            # 不分配虚拟 IP
  selector:
    app: mysql
```

DNS 直接返回所有后端 Pod 的 IP 列表（而不是虚拟 IP），常用于 StatefulSet：

```
# DNS 查询返回所有 Pod IP
$ nslookup mysql-headless
Name: 10.244.1.5
Name: 10.244.2.8
Name: 10.244.3.2

# 也可以查询单个 Pod
$ nslookup mysql-0.mysql-headless
Name: 10.244.1.5
```

### NodePort

```yaml
apiVersion: v1
kind: Service
metadata:
  name: webapp-nodeport
spec:
  type: NodePort
  selector:
    app: webapp
  ports:
  - port: 80                  # Service 端口（集群内部访问）
    targetPort: 8080           # Pod 实际端口
    nodePort: 30080            # 每个 Node 上暴露的端口
```

```
外部访问方式：
  curl http://<任意Node-IP>:30080
```

### LoadBalancer

```yaml
apiVersion: v1

  name: webapp-lb

  type: LoadBalancer

    app: webapp

  - port: 80
```

### kube-proxy、DNS、CNI 的职责边界

初学者最容易混淆三个网络组件的分工：**kube-proxy**、**CoreDNS**、**CNI（Calico）**。关键在于它们各自处在数据包生命周期的不同环节，谁也不替代谁。

**一次完整请求经过的三个环节：**

```
Pod 发起请求：curl http://webapp-svc.default.svc.cluster.local

  ① DNS 解析（CoreDNS 管）
     webapp-svc... → 解析为 ClusterIP 10.96.0.10
     （这一步 kube-proxy 完全不参与，它不"解析"任何东西）

  ② 包转发改写（kube-proxy 管的规则 + 内核执行）
     Pod 发出包，目的 IP = 10.96.0.10
     内核 netfilter/IPVS 匹配到 kube-proxy 写的规则
     → DNAT：目的地址改成后端 Pod IP 10.244.1.5

  ③ 路由送达（CNI/Calico 管）
     包按 Pod IP 10.244.1.5 路由
     同节点 → veth pair 直达
     跨节点 → Calico BGP/VXLAN 路由到目标 Node
```

**职责对照表：**

| 组件 | 解决的问题 | 操作对象 | 工作层 |
|------|-----------|---------|--------|
| **CoreDNS** | 名字 → IP（解析） | Service 域名 / Pod 域名 | 应用层（DNS 协议） |
| **kube-proxy** | Service IP → Pod IP（转发改写） | ClusterIP / NodePort / LB VIP | 内核 netfilter / IPVS（L3/L4） |
| **CNI（Calico）** | Pod IP 的分配与可达性 | Pod IP、veth pair、路由表 | 网络层 / 数据链路层 |

**三个容易踩坑的认知：**

1. **kube-proxy 不管 Pod IP。** Pod IP 是 CNI 分配的真实可路由地址，Pod 间直接用 Pod IP 通信走的是 Calico 的 BGP/VXLAN 路由，完全不碰 kube-proxy。换句话说：kube-proxy 管 **Service → Pod** 的转发（服务发现抽象），Calico 管 **Pod → Pod** 的底层可达性，两者职责正交。

2. **Headless Service 绕过 kube-proxy。** 当 `clusterIP: None` 时，CoreDNS 直接返回所有后端 Pod IP（不再返回单个 ClusterIP），客户端自己挑一个 Pod IP 直连。整条链路里 kube-proxy 一条规则都不写。这也是 StatefulSet 用 Headless Service 的原因——客户端需要直连特定 Pod（如 `mysql-0`），而非随机负载均衡。

3. **kube-proxy 本身不在数据路径上。** 它叫 "proxy" 但**不转发流量**，只是个"规则编程器"——watch apiserver 上的 Service/EndpointSlice 变化，把规则写进内核。真正改包的是 Linux 内核的 netfilter/IPVS。因此：
   - kube-proxy 进程挂了 → 已有规则仍在，旧流量继续转发，但新建 Service / 扩缩容的规则不会更新
   - 规则一旦写好，转发在内核态完成，延迟极低，不经过任何用户态进程

**一句话记忆：** DNS 管解析名字、kube-proxy 管转发 Service 地址、CNI 管路由 Pod IP——三者接力，互不替代。

## 4.3 Ingress

Ingress 提供 HTTP/HTTPS 路由规则，将外部流量路由到不同的 Service。它工作在 L7（应用层），比 Service（L4）提供更丰富的路由能力。

### Ingress Controller

Ingress 资源需要配合 **Ingress Controller** 才能工作。常见的 Controller：

| Controller | 说明 |
|-----------|------|
| **NGINX Ingress** | 最常用，功能完善 |
| **Traefik** | 自动服务发现，配置简单 |
| **Kong** | 基于 OpenResty，插件丰富 |
| **AWS ALB Controller** | AWS 原生 ALB 集成 |

### 基础路由

```yaml
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: webapp-ingress
  annotations:
    nginx.ingress.kubernetes.io/rewrite-target: /
spec:
  ingressClassName: nginx          # 指定使用哪个 Ingress Controller
  rules:
  - host: app.example.com
    http:
      paths:
      - path: /api
        pathType: Prefix
        backend:
          service:
            name: api-svc
            port:
              number: 80
      - path: /
        pathType: Prefix
        backend:
          service:
            name: frontend-svc
            port:
              number: 80
```

```
请求路由：
  app.example.com/api/*    → api-svc:80
  app.example.com/*        → frontend-svc:80
```

### TLS/HTTPS

```yaml
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: secure-ingress
spec:
  ingressClassName: nginx
  tls:
  - hosts:
    - app.example.com
    secretName: tls-secret           # 引用 TLS 证书 Secret
  rules:
  - host: app.example.com
    http:
      paths:
      - path: /
        pathType: Prefix
        backend:
          service:
            name: webapp-svc
            port:
              number: 443
```

创建 TLS Secret：

```bash
kubectl create secret tls tls-secret \
  --cert=tls.crt --key=tls.key
```

### 高级配置

```yaml
metadata:
  annotations:
    # 限流
    nginx.ingress.kubernetes.io/limit-rps: "10"
    # CORS
    nginx.ingress.kubernetes.io/enable-cors: "true"
    # 认证
    nginx.ingress.kubernetes.io/auth-type: basic
    nginx.ingress.kubernetes.io/auth-secret: basic-auth
    # WebSocket 支持
    nginx.ingress.kubernetes.io/proxy-read-timeout: "3600"
    nginx.ingress.kubernetes.io/proxy-send-timeout: "3600"

    # 请求体大小
    nginx.ingress.kubernetes.io/proxy-body-size: "50m"
```

## 4.4 DNS（CoreDNS）

K8s 内置 DNS 服务（CoreDNS），为 Service 和 Pod 提供域名解析。

### DNS 记录格式

```
# Service DNS
<service-name>.<namespace>.svc.cluster.local

# Pod DNS（需要 StatefulSet + Headless Service）
<pod-name>.<headless-svc-name>.<namespace>.svc.cluster.local

# 示例
webapp-svc.default.svc.cluster.local     → Service ClusterIP
mysql-0.mysql-headless.prod.svc.cluster.local → 特定 Pod IP
```

### DNS 查询流程

```
Pod 发起 DNS 查询
  │
  ├─ /etc/resolv.conf 配置
  │   nameserver 10.96.0.10      ← CoreDNS Service IP
  │   search default.svc.cluster.local svc.cluster.local cluster.local
  │
  └─→ CoreDNS 解析
       ├─ K8s Service → 返回 ClusterIP
       ├─ K8s Headless Service → 返回 Pod IP 列表
       └─ 外部域名 → 转发到上游 DNS（如 8.8.8.8）
```

### CoreDNS 配置（Corefile）

```bash
# 查看 CoreDNS 配置
kubectl get configmap coredns -n kube-system -o yaml
```

```yaml
# Corefile 核心配置
.:53 {
    errors                          # 错误日志
    health                          # 健康检查端点
    ready                           # 就绪端点
    kubernetes cluster.local in-addr.arpa ip6.arpa {
       pods insecure                # Pod DNS 记录策略
       fallthrough in-addr.arpa ip6.arpa
    }
    prometheus :9153                # 指标暴露
    forward . /etc/resolv.conf      # 外部域名转发
    cache 30                        # 缓存 30 秒
    loop                            # 检测转发循环
    reload                          # 自动重载配置
    loadbalance                     # 轮询负载均衡
}
```

### CoreDNS 端口

| 协议 | 端口 | 说明 |
|------|------|------|
| **UDP/TCP** | **53** | DNS 服务端口。集群内所有 Pod 的 DNS 查询都发往 CoreDNS 的 `10.96.0.10:53`（kube-dns Service） |
| **TCP** | **9153** | CoreDNS Prometheus 指标端口。暴露 DNS 查询量、缓存命中率、延迟等指标供监控系统采集 |

### NodeLocal DNSCache — DNS 性能优化

默认 DNS 架构在大规模集群下有明显瓶颈。Pod 的 `/etc/resolv.conf` 指向 kube-dns 的 ClusterIP（`10.96.0.10:53`），每次查询都要走 Service 的转发链路：

```
默认架构：
  Pod → 10.96.0.10:53 (kube-dns ClusterIP)
       → iptables/IPVS 规则匹配 → DNAT
       → 某个 CoreDNS Pod（可能跨节点）
```

**三个痛点：**

1. **conntrack 压力**：DNS 默认走 UDP，每次查询都产生一条 conntrack 表项。DNS 是短小高频请求，大规模集群下 conntrack 表会暴涨，甚至打满导致丢包
2. **DNAT 开销**：每条查询都要过 iptables 规则匹配 + DNAT，高 QPS 下成为 CPU 瓶颈
3. **跨节点延迟**：CoreDNS Pod 调度到别的节点，查询要绕一段网络

**NodeLocal DNSCache 方案：** 在每个 Node 上以 DaemonSet（`hostNetwork: true`）运行一个本地 DNS 缓存实例（本质也是 CoreDNS + cache 插件），监听一个 **本地 link-local 地址 `169.254.20.10`**。这个地址属于 `169.254.0.0/16` 链路本地段，只在单节点有效，不会与集群任何 Pod/Service IP 冲突。

```
NodeLocal DNSCache 架构：

  ┌─────────────────────────────────────────────┐
  │                  Node-1                     │
  │                                             │
  │   Pod A            Pod B                    │
  │   resolv.conf:     resolv.conf:             │
  │   169.254.20.10    169.254.20.10   ← 指向本地 │
  │       │                │                    │
  │       └───────┬────────┘                    │
  │               ▼  （本机地址，不走 iptables）  │
  │   ┌────────────────────────┐                │
  │   │ NodeLocal DNSCache     │ ← 本地缓存命中  │
  │   │ 169.254.20.10:53       │   直接返回      │
  │   └───────────┬────────────┘                │
  └───────────────┼─────────────────────────────┘
                  ▼ 缓存未命中
          kube-dns ClusterIP (10.96.0.10)
                  ▼
              CoreDNS Pod
```

**查询转发逻辑（Corefile 配置核心）：**

```yaml
# NodeLocal DNSCache 的 Corefile（节选）
# 1. 集群内域名：回源到 kube-dns（CoreDNS）
cluster.local:53 {
    errors
    cache {                # 本地缓存
        success 9984 30    # 成功响应缓存 30s
        denial 9984 5      # 否定响应缓存 5s
    }
    bind 169.254.20.10
    forward . __PILLAR__DNS__SERVER__ {   # → kube-dns ClusterIP
        force_tcp          # 用 TCP 转发，避免 UDP conntrack
    }
    prometheus :9253
}

# 2. 集群外域名：直接转发到上游 DNS（不经过 kube-dns）
.:53 {
    errors
    cache 30
    bind 169.254.20.10
    forward . __PILLAR__UPSTREAM__SERVERS__   # → 节点 /etc/resolv.conf 上游
    prometheus :9253
}
```

**核心收益：**

| 优化点 | 原理 |
|--------|------|
| **绕过 conntrack** | 访问本地 `169.254.20.10` 是本机地址，不进 Service DNAT 链，不产生 conntrack 表项 |
| **绕过 iptables** | 本地地址不匹配 KUBE-SERVICES 规则，省掉规则匹配开销 |
| **本地缓存** | 命中即返回，纯本机内存操作，无网络往返 |
| **TCP 回源** | `force_tcp` 让 NodeLocal → CoreDNS 走 TCP，进一步减少 UDP conntrack |

**部署步骤：**

```bash
# 1. 获取官方 manifest
wget https://raw.githubusercontent.com/kubernetes/kubernetes/master/cluster/addons/dns/nodelocaldns/nodelocaldns.yaml

# 2. 替换占位符变量
#    __PILLAR__LOCAL__DNS__         → 169.254.20.10  (本地监听地址)
#    __PILLAR__DNS__DOMAIN__        → cluster.local
#    __PILLAR__DNS__SERVER__        → 10.96.0.10     (kube-dns ClusterIP)
#    __PILLAR__UPSTREAM__SERVERS__  → /etc/resolv.conf

# 3. 部署（DaemonSet，每个 Node 一个，hostNetwork: true）
kubectl apply -f nodelocaldns.yaml

# 4. 修改 kubelet 的 clusterDNS 指向本地地址（让新 Pod 用上）
#    /var/lib/kubelet/config.yaml:
#    clusterDNS:
#    - 169.254.20.10
systemctl restart kubelet

# 5. 重建已有 Pod，使其 resolv.conf 指向新地址
kubectl delete pods --all  # 滚动重启更稳妥
```

**两种监听模式对比：**

| 模式 | 监听地址 | 是否需改 kubelet | 特点 |
|------|---------|----------------|------|
| **Local IP 模式**（默认） | `169.254.20.10` | 是（改 clusterDNS + 重建 Pod） | 通用、推荐，无 IP 冲突风险 |
| **ClusterIP 模式** | kube-dns ClusterIP `10.96.0.10` | 否（Pod 无感迁移） | 需 IPVS 配合，无缝接入但配置复杂 |

**注意事项：**

1. **单点风险**：标准模式下，某节点的 NodeLocal DNSCache 进程挂掉，该节点所有 Pod 的 DNS 查询会直接失败（指向的本地地址无进程监听）。DaemonSet 会自动重启，但必须配监控告警
2. **CoreDNS 不能省略**：集群内域名仍需回源到 CoreDNS 解析，NodeLocal DNSCache 只是缓存层，不是替代品
3. **`force_tcp` 的权衡**：TCP 转发省 conntrack 但有握手开销，极低延迟场景需权衡
4. **`hostNetwork: true` 是必须的**：只有用宿主机网络栈，才能绑定 `169.254.20.10` 并让 Pod 当本地地址访问

> **适用场景**：节点数 > 50、Pod 密度高、DNS QPS 大的集群收益最明显；小集群收益有限，可不启用。

## 4.5 NetworkPolicy

NetworkPolicy 控制 Pod 之间的网络流量，实现网络隔离。需要 CNI 插件支持（Calico、Cilium 等，Flannel 默认不支持）。

### 默认行为

- 没有 NetworkPolicy 时：所有 Pod 可以互相通信（无限制）
- 一旦为某个 Pod 创建了 NetworkPolicy：只允许策略中定义的流量

### 示例：限制 Web 只能被 API 访问

```yaml
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: web-access
  namespace: default
spec:
  podSelector:
    matchLabels:
      app: web              # 作用于 app=web 的 Pod
  policyTypes:
  - Ingress                 # 入站规则
  ingress:
  - from:
    - podSelector:
        matchLabels:
          app: api           # 只允许 app=api 的 Pod 访问
    ports:
    - protocol: TCP
      port: 8080
```

### 示例：隔离 namespace

```yaml
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: deny-other-namespaces
  namespace: production
spec:
  podSelector: {}            # 作用于该 namespace 下所有 Pod
  policyTypes:
  - Ingress
  ingress:
  - from:
    - namespaceSelector:
        matchLabels:
          name: production   # 只允许同 namespace
    - podSelector: {}        # 同 namespace 的所有 Pod
```

### 示例：允许出站但限制目标

```yaml
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: restrict-egress
spec:
  podSelector:
    matchLabels:
      app: backend
  policyTypes:
  - Egress
  egress:
  - to:
    - podSelector:
        matchLabels:
          app: database
    ports:
    - protocol: TCP
      port: 3306             # 只能访问 MySQL
  - to:
    - namespaceSelector: {}
    ports:
    - protocol: UDP
      port: 53               # 允许 DNS 查询
```

|------|-----------|---------|---------|
| CNI (Calico) | Pod 网络基石，负责 IP 分配和跨节点通信 | TCP 179 (BGP), 5473 (Typha), UDP 4789 (VXLAN) | veth pair, BGP, VXLAN, IPAM |

| DNS (CoreDNS) | 集群内域名解析 | UDP/TCP 53, TCP 9153 (metrics) | service.namespace.svc.cluster.local |
| NetworkPolicy | Pod 间网络隔离 | — | Ingress/Egress 规则, CNI 依赖 |
