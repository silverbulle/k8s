# 04 - 服务与网络

K8s 的网络模型解决了三个层次的问题：**Pod 之间如何通信**（CNI 插件实现）、**集群内服务发现**（Service + DNS）、**外部如何访问集群**（Ingress）。本章从底层网络到上层服务逐层展开。

## K8s 网络基础

K8s 的网络设计遵循以下原则：

1. **每个 Pod 拥有独立 IP**（不需要 NAT）
2. **所有 Pod 之间可以直接通信**（扁平网络，无防火墙限制）
3. **Node 可以与该 Node 上的所有 Pod 通信**

```
┌──────────────────────────────────────────────┐
│              K8s Cluster                      │
│                                              │
│  Node-1                   Node-2             │
│  ┌─────────┐ ┌─────────┐ ┌─────────┐        │
│  │ Pod A   │ │ Pod B   │ │ Pod C   │        │
│  │10.244   │ │10.244   │ │10.244   │        │
│  │ .1.5    │◄┤ .1.8    │◄┤ .2.3    │        │
│  └─────────┘ └─────────┘ └─────────┘        │
│                                              │
│  Service: 10.96.0.10 (虚拟 IP，通过 iptables 转发)
└──────────────────────────────────────────────┘
```

**Pod IP 是临时的**——Pod 重建后 IP 会变。Service 提供了稳定的访问入口。

## 4.1 CNI 与 Pod 网络

K8s 自身不实现 Pod 网络，而是通过 **CNI（Container Network Interface）** 规范将网络实现委托给第三方插件。CNI 插件负责：Pod IP 分配、跨节点 Pod 通信、NetworkPolicy 执行。

### CNI 工作流程

```
kubelet 创建 Pod
  │
  ├─ 1. 调用 CRI 创建 Pod Sandbox（Pause 容器）
  │
  ├─ 2. kubelet 调用 CNI 插件
  │     ┌───────────────────────────────────────────┐
  │     │  CNI ADD 请求：                           │
  │     │  - Container ID                          │
  │     │  - Network Namespace (Pause 容器的 ns)    │
  │     │  - 接口名: eth0                           │
  │     └───────────────────────────────────────────┘
  │
  ├─ 3. CNI 插件执行：
  │     - 分配 IP 地址（IPAM）
  │     - 配置网络接口（veth pair）
  │     - 设置路由规则
  │     - 将 Pod 网络接入集群网络
  │
  └─ 4. 返回结果给 kubelet：
        { "ip4": { "ip": "10.244.1.5/24", "gateway": "10.244.1.1" } }
```

### CNI 配置文件

kubelet 启动时读取 CNI 配置（通常在 `/etc/cni/net.d/`）：

```json
// /etc/cni/net.d/10-calico.conflist
{
  "name": "k8s-pod-network",
  "cniVersion": "0.3.1",
  "plugins": [
    {
      "type": "calico",
      "log_level": "info",
      "datastore_type": "kubernetes",
      "mtu": 1440,
      "ipam": {
        "type": "calico-ipam"
      },
      "policy": {
        "type": "k8s"
      },
      "kubernetes": {
        "kubeconfig": "/etc/cni/net.d/calico-kubeconfig"
      }
    },
    {
      "type": "portmap",
      "snat": true,
      "capabilities": {"portMappings": true}
    },
    {
      "type": "bandwidth",
      "capabilities": {"bandwidth": true}
    }
  ]
}
```

### veth pair — Pod 网络的基石

每个 Pod 通过一对虚拟以太网设备（veth pair）连接到宿主机网络：

```
Node-1
┌──────────────────────────────────────────────────┐
│                                                  │
│  ┌────────────────┐        ┌────────────────┐   │
│  │ Pod A          │        │ Pod B          │   │
│  │ ┌────────────┐ │        │ ┌────────────┐ │   │
│  │ │  eth0      │ │        │ │  eth0      │ │   │
│  │ │ 10.244.1.5 │ │        │ │ 10.244.1.8 │ │   │
│  │ └──────┬─────┘ │        │ └──────┬─────┘ │   │
│  └────────┼────────┘        └────────┼────────┘   │
│           │ veth pair               │ veth pair   │
│  ┌────────┴─────┐        ┌────────┴─────┐       │
│  │ cali1234...  │        │ cali5678...  │       │
│  └────────┬─────┘        └────────┬─────┘       │
│           └────────────┬──────────┘              │
│                        ▼                         │
│           ┌──────────────────────┐               │
│           │   宿主机路由表        │               │
│           │   或 BGP/VXLAN 转发  │               │
│           └──────────────────────┘               │
└──────────────────────────────────────────────────┘
```

## Calico 架构

Calico 是生产环境最常用的 CNI 插件之一，由 Project Calico 开发，Tigera 维护。它使用纯三层（L3）网络方案，不依赖 overlay（可选 VXLAN）。

### 核心组件

```
┌────────────────────────────────────────────────────────────┐
│                    Calico 组件总览                           │
│                                                            │
│  每个 Node 上运行（DaemonSet）:                              │
│  ┌──────────────────────────────────────────────────┐      │
│  │  calico-node                                      │      │
│  │  ├── Felix         — 编程路由规则和 iptables 策略  │      │
│  │  ├── BIRD          — BGP 路由协议守护进程          │      │
│  │  └── confd         — 从 datastore 生成 BIRD 配置  │      │
│  └──────────────────────────────────────────────────┘      │
│                                                            │
│  集群级运行（Deployment）:                                    │
│  ┌──────────────────────────────────────────────────┐      │
│  │  calico-kube-controllers                          │      │
│  │  ├── Node Controller      — Node 生命周期管理      │      │
│  │  ├── Policy Controller    — NetworkPolicy 同步     │      │
│  │  ├── IPAM Controller      — IP 地址管理            │      │
│  │  └── ServiceAccount Ctrl  — SA Token 管理         │      │
│  └──────────────────────────────────────────────────┘      │
│                                                            │
│  可选组件:                                                  │
│  ┌──────────────────────────────────────────────────┐      │
│  │  Typha           — Felix/BIRD 与 datastore 的代理 │      │
│  │  calico-typha    — 大规模集群减少 apiserver 压力   │      │
│  └──────────────────────────────────────────────────┘      │
└────────────────────────────────────────────────────────────┘
```

### Felix — 数据平面代理

Felix 运行在每个 Node 上，是 Calico 的核心数据平面组件：

| 职责 | 说明 |
|------|------|
| **路由编程** | 将 Pod IP 路由写入宿主机路由表 |
| **iptables/nftables 规则** | 实现 NetworkPolicy（入站/出站过滤） |
| **接口管理** | 管理 veth pair 的创建和配置 |
| **ARP/NDP 代理** | 响应 ARP 请求，让同节点 Pod 可通信 |
| **健康检查** | 监控网络连通性，上报 Node 状态 |

```
Felix 的数据流：

  Datastore (etcd/K8s API)
       │
       │ Watch 资源变更
       ▼
  ┌─────────┐
  │  Felix  │ ──► 写入宿主机路由表（ip route）
  └─────────┘ ──► 写入 iptables 规则（NetworkPolicy）
            ──► 管理 veth pair
            ──► 响应 ARP 请求
```

### BIRD — BGP 路由协议

BIRD 在 Node 之间通过 **BGP（Border Gateway Protocol）** 交换路由信息，让每个 Node 知道其他 Node 上的 Pod IP 段：

```
Node-1 (Pod CIDR: 10.244.1.0/24)          Node-2 (Pod CIDR: 10.244.2.0/24)
┌──────────────────────┐                   ┌──────────────────────┐
│  BIRD                │   BGP Session     │  BIRD                │
│                      │◄────────────────►│                      │
│  宣告:               │   TCP:179         │  宣告:               │
│  10.244.1.0/24       │                   │  10.244.2.0/24       │
│  via 192.168.1.10    │                   │  via 192.168.1.11    │
└──────────────────────┘                   └──────────────────────┘

路由结果（Node-1 的路由表）：
  10.244.2.0/24 via 192.168.1.11    ← 通过 BGP 学到
  10.244.1.0/24 dev calixxx         ← 本地直连
```

**BGP 模式对比：**

| 模式 | 说明 | 适用场景 |
|------|------|---------|
| **Full Mesh** | 每个 Node 与其他所有 Node 建立 BGP Session | 中小集群（< 100 Node） |
| **Route Reflector** | 指定 RR 节点集中分发路由，减少 BGP Session 数 | 大规模集群 |
| **AS Per Rack** | 与物理路由器 BGP Peering，融入数据中心网络 | 物理网络集成 |

```yaml
# 配置 BGP Peer（Route Reflector 模式）
apiVersion: projectcalico.org/v3
kind: BGPPeer
metadata:
  name: route-reflector
spec:
  peerIP: 192.168.1.1
  asNumber: 64512
  nodeSelector: all()
```

### calico-kube-controllers

calico-kube-controllers 是集群级的控制平面组件，包含多个子控制器：

| 控制器 | 职责 |
|--------|------|
| **Node Controller** | 监听 K8s Node 事件，同步创建/删除 Calico Node 资源；Node 删除时清理该 Node 上的 IPAM 分配 |
| **Policy Controller** | 监听 K8s NetworkPolicy，转换为 Calico NetworkPolicy 格式供 Felix 消费 |
| **IPAM Controller** | 管理 IP 地址分配，处理 Node 删除时的 IP 回收 |
| **ServiceAccount Controller** | 同步 K8s ServiceAccount 到 Calico 策略 |

### Calico IPAM — IP 地址管理

```
IP 分配流程：

1. kubelet 创建 Pod → 调用 CNI
2. Calico CNI 请求 IPAM 分配 IP
3. IPAM 从 IPPool 中分配：
   - 优先从当前 Node 的 Pod CIDR 中分配
   - Node CIDR 耗尽时从全局 Pool 分配
4. IP 与 Pod 绑定记录到 datastore（K8s CRD 或 etcd）

IPPool 配置：
```

```yaml
apiVersion: projectcalico.org/v3
kind: IPPool
metadata:
  name: default-ipv4-pool
spec:
  cidr: 10.244.0.0/16            # Pod 地址池
  blockSize: 26                   # 每个 Node 分配 /26（64 个 IP）
  ipipMode: Never                 # 不使用 IPIP 封装
  vxlanMode: Never                # 不使用 VXLAN 封装
  natOutgoing: true               # Pod 访问外部时做 SNAT
  nodeSelector: all()
```

## Calico 网络模式

Calico 支持三种跨节点 Pod 通信模式：

### 模式一：BGP 纯三层路由（推荐）

```
Node-1                                  Node-2
┌──────────────┐                        ┌──────────────┐
│  Pod A       │                        │  Pod C       │
│  10.244.1.5  │                        │  10.244.2.3  │
└──────┬───────┘                        └──────┬───────┘
       │ veth                                  │ veth
       ▼                                       ▼
  宿主机路由表                              宿主机路由表
  10.244.2.0/24 via Node-2 IP            10.244.1.0/24 via Node-1 IP
       │                                       │
       └──── BGP 宣告路由 ──────────────────────┘
              （原始 IP 包，无封装）
```

- **优点**：性能最好（无封装开销），网络可观测（tcpdump 直接抓原始包）
- **要求**：底层网络支持路由传播（二层可达或 BGP Peering）
- **适用**：裸金属、VM 同子网、云环境 VPC 内

### 模式二：VXLAN 封装

```
Node-1                                  Node-2
┌──────────────┐                        ┌──────────────┐
│  Pod A       │                        │  Pod C       │
│  10.244.1.5  │                        │  10.244.2.3  │
└──────┬───────┘                        └──────┬───────┘
       │                                       │
       ▼                                       ▼
  ┌─────────────────────────────────────────────────┐
  │  VXLAN 封装                                     │
  │  原始 IP 包 → 加 UDP/VXLAN 头 → 外层 Node IP    │
  │  src: Node-1 IP  →  dst: Node-2 IP              │
  └─────────────────────────────────────────────────┘
```

- **优点**：不依赖底层路由，跨子网/跨 AZ 可用
- **缺点**：封装开销（约 5-10% 性能损失），MTU 需要减小 50 字节
- **适用**：跨子网、云环境跨 AZ

### 模式三：IPIP 封装

```
类似 VXLAN，但使用 IP-in-IP 封装（协议号 4）
- 开销比 VXLAN 小（20 字节 vs 50 字节）
- 部分云环境不支持（如 AWS 丢弃 IPIP 包）
- 逐渐被 VXLAN 取代
```

```yaml
# 切换网络模式（修改 IPPool）
spec:
  ipipMode: Never        # Never | Always | CrossSubnet
  vxlanMode: Always      # Never | Always | CrossSubnet

# CrossSubnet = 同子网用 BGP，跨子网用封装（最佳实践）
```

## Calico NetworkPolicy 增强

Calico 扩展了 K8s 原生 NetworkPolicy，提供更多能力：

```yaml
# Calico 特有的策略能力
apiVersion: projectcalico.org/v3
kind: NetworkPolicy
metadata:
  name: advanced-policy
  namespace: production
spec:
  selector: app == 'web'              # 表达式选择器（比 matchLabels 更灵活）
  types:
  - Ingress
  - Egress
  ingress:
  - action: Allow
    source:
      selector: role == 'frontend'
      namespaceSelector: name == 'web'  # 跨 namespace 选择
    destination:
      ports:
      - 80
      - 443
  egress:
  - action: Allow
    destination:
      domains:                         # 域名级出站控制（K8s 原生不支持）
      - '*.example.com'
      - 'api.github.com'
      ports:
      - 443
  - action: Deny                        # 显式拒绝（K8s 原生只有 Allow）
    destination:
      nets:
      - 169.254.169.254/32             # 禁止访问云元数据服务
```

**Calico vs K8s NetworkPolicy 对比：**

| 特性 | K8s NetworkPolicy | Calico NetworkPolicy |
|------|-------------------|---------------------|
| 动作 | 只有 Allow | Allow / Deny / Log |
| 选择器 | matchLabels | 表达式（==, !=, in, not in） |
| 跨 Namespace | namespaceSelector | namespaceSelector + 表达式 |
| 域名过滤 | 不支持 | 支持 `domains` |
| 全局策略 | 不支持 | GlobalNetworkPolicy |
| 优先级 | 无（全部 Allow） | order 字段排序 |
| 日志 | 不支持 | action: Log |

## 主流 CNI 方案对比

| 特性 | Calico | Flannel | Cilium | Weave |
|------|--------|---------|--------|-------|
| **网络模式** | BGP / VXLAN / IPIP | VXLAN | eBPF / VXLAN | Weave Mesh |
| **数据平面** | iptables / BPF | VXLAN | eBPF（XDP/TC） | iptables |
| **NetworkPolicy** | 支持（增强） | 不支持 | 支持（增强） | 支持 |
| **性能** | 高（BGP 无封装） | 中（VXLAN 开销） | 极高（eBPF） | 中 |
| **可观测性** | 一般 | 基本 | 强（Hubble） | 一般 |
| **大规模集群** | 支持（Typha + RR） | 不推荐 | 推荐 | 不推荐 |
| **加密** | WireGuard | 不支持 | IPsec / WireGuard | 支持 |
| **维护方** | Tigera | CoreOS（已停更） | Isovalent | Weaveworks |

**选择建议：**
- **通用生产**：Calico（成熟稳定，社区支持好）
- **高性能/可观测**：Cilium（eBPF 数据平面，Hubble 可观测性）
- **简单测试**：Flannel（配置最简单，功能最少）

## 4.2 Service

Service 是一组 Pod 的逻辑抽象，提供稳定的网络端点（IP + 端口），后端 Pod 变化时自动更新。

### Service 类型

| 类型 | 访问范围 | 实现方式 |
|------|---------|---------|
| **ClusterIP**（默认） | 集群内部 | 分配虚拟 IP，通过 iptables/IPVS 转发 |
| **NodePort** | 集群外部 | 在每个 Node 上开放固定端口（30000-32767） |
| **LoadBalancer** | 集群外部 | 云厂商负载均衡器 |
| **ExternalName** | 外部服务 | CNAME 映射到外部域名 |

### ClusterIP

```yaml
apiVersion: v1
kind: Service
metadata:
  name: webapp-svc
spec:
  type: ClusterIP
  selector:
    app: webapp              # 匹配带有 app=webapp 标签的 Pod
  ports:
  - port: 80                 # Service 端口
    targetPort: 8080          # Pod 实际端口
    protocol: TCP
```

```
集群内访问方式：
  curl http://webapp-svc.default.svc.cluster.local:80
  curl http://webapp-svc:80              # 同 namespace 可缩写
```

**Headless Service**（ClusterIP: None）：

```yaml
spec:
  clusterIP: None            # 不分配虚拟 IP
  selector:
    app: mysql
```

DNS 直接返回所有后端 Pod 的 IP 列表（而不是虚拟 IP），常用于 StatefulSet：

```
# DNS 查询返回所有 Pod IP
$ nslookup mysql-headless
Name: 10.244.1.5
Name: 10.244.2.8
Name: 10.244.3.2

# 也可以查询单个 Pod
$ nslookup mysql-0.mysql-headless
Name: 10.244.1.5
```

### NodePort

```yaml
apiVersion: v1
kind: Service
metadata:
  name: webapp-nodeport
spec:
  type: NodePort
  selector:
    app: webapp
  ports:
  - port: 80                  # Service 端口（集群内部访问）
    targetPort: 8080           # Pod 实际端口
    nodePort: 30080            # 每个 Node 上暴露的端口
```

```
外部访问方式：
  curl http://<任意Node-IP>:30080
```

### LoadBalancer

```yaml
apiVersion: v1
kind: Service
metadata:
  name: webapp-lb
spec:
  type: LoadBalancer
  selector:
    app: webapp
  ports:
  - port: 80
    targetPort: 8080
```

在云环境中，K8s 会自动创建一个云负载均衡器（AWS ELB/ALB、GCP LB、Azure LB）并将流量引入。

```bash
# 查看分配的外部 IP
kubectl get svc webapp-lb
NAME          TYPE           EXTERNAL-IP     PORT(S)
webapp-lb     LoadBalancer   52.1.2.3        80:31234/TCP
```

### ExternalName

```yaml
apiVersion: v1
kind: Service
metadata:
  name: external-db
spec:
  type: ExternalName
  externalName: db.example.com    # CNAME 映射
```

```
# 集群内访问 external-db → DNS 解析为 db.example.com
curl http://external-db.default.svc.cluster.local
# 实际访问 → http://db.example.com
```

### Service 与 Endpoints

```
Service (selector: app=webapp)
  │
  ├──► Endpoints 对象（自动维护）
  │      ├── 10.244.1.5:8080
  │      ├── 10.244.2.8:8080
  │      └── 10.244.3.2:8080
  │
  └── 只有 Readiness Probe 通过的 Pod 才会出现在 Endpoints 中
```

```bash
# 查看 Endpoints

webapp-svc    10.244.1.5:8080,10.244.2.8:8080,...        5m
```

## 4.3 Ingress

Ingress 提供 HTTP/HTTPS 路由规则，将外部流量路由到不同的 Service。它工作在 L7（应用层），比 Service（L4）提供更丰富的路由能力。

### Ingress Controller

Ingress 资源需要配合 **Ingress Controller** 才能工作。常见的 Controller：

| Controller | 说明 |
|-----------|------|
| **NGINX Ingress** | 最常用，功能完善 |
| **Traefik** | 自动服务发现，配置简单 |
| **Kong** | 基于 OpenResty，插件丰富 |
| **AWS ALB Controller** | AWS 原生 ALB 集成 |

### 基础路由

```yaml
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: webapp-ingress
  annotations:
    nginx.ingress.kubernetes.io/rewrite-target: /
spec:
  ingressClassName: nginx          # 指定使用哪个 Ingress Controller
  rules:
  - host: app.example.com
    http:
      paths:
      - path: /api
        pathType: Prefix
        backend:
          service:
            name: api-svc
            port:
              number: 80
      - path: /
        pathType: Prefix
        backend:
          service:
            name: frontend-svc
            port:
              number: 80
```

```
请求路由：
  app.example.com/api/*    → api-svc:80
  app.example.com/*        → frontend-svc:80
```

## 4.4 DNS（CoreDNS）

K8s 内置 DNS 服务（CoreDNS），为 Service 和 Pod 提供域名解析。

### DNS 记录格式

```
# Service DNS
<service-name>.<namespace>.svc.cluster.local

# Pod DNS（需要 StatefulSet + Headless Service）
<pod-name>.<headless-svc-name>.<namespace>.svc.cluster.local

# 示例
webapp-svc.default.svc.cluster.local     → Service ClusterIP
mysql-0.mysql-headless.prod.svc.cluster.local → 特定 Pod IP
```

### DNS 查询流程

```
Pod 发起 DNS 查询
  │
  ├─ /etc/resolv.conf 配置
  │   nameserver 10.96.0.10      ← CoreDNS Service IP
  │   search default.svc.cluster.local svc.cluster.local cluster.local
  │
  └─→ CoreDNS 解析
       ├─ K8s Service → 返回 ClusterIP
       ├─ K8s Headless Service → 返回 Pod IP 列表
       └─ 外部域名 → 转发到上游 DNS（如 8.8.8.8）
```

### CoreDNS 配置（Corefile）

```bash
# 查看 CoreDNS 配置
kubectl get configmap coredns -n kube-system -o yaml
```

```yaml
# Corefile 核心配置
.:53 {
    errors                          # 错误日志
    health                          # 健康检查端点
    ready                           # 就绪端点
    kubernetes cluster.local in-addr.arpa ip6.arpa {
       pods insecure                # Pod DNS 记录策略
       fallthrough in-addr.arpa ip6.arpa
    }
    prometheus :9153                # 指标暴露
    forward . /etc/resolv.conf      # 外部域名转发
    cache 30                        # 缓存 30 秒
    loop                            # 检测转发循环
    reload                          # 自动重载配置
    loadbalance                     # 轮询负载均衡
}
```

## 4.5 NetworkPolicy

NetworkPolicy 控制 Pod 之间的网络流量，实现网络隔离。需要 CNI 插件支持（Calico、Cilium 等，Flannel 默认不支持）。

### 默认行为

- 没有 NetworkPolicy 时：所有 Pod 可以互相通信（无限制）
- 一旦为某个 Pod 创建了 NetworkPolicy：只允许策略中定义的流量

### 示例：限制 Web 只能被 API 访问

```yaml
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: web-access
  namespace: default
spec:
  podSelector:
    matchLabels:
      app: web              # 作用于 app=web 的 Pod
  policyTypes:
  - Ingress                 # 入站规则
  ingress:
  - from:
    - podSelector:
        matchLabels:
          app: api           # 只允许 app=api 的 Pod 访问
    ports:
    - protocol: TCP
      port: 8080
```

### 示例：隔离 namespace

```yaml
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: deny-other-namespaces
  namespace: production
spec:
  podSelector: {}            # 作用于该 namespace 下所有 Pod
  policyTypes:
  - Ingress
  ingress:
  - from:
    - namespaceSelector:
        matchLabels:
          name: production   # 只允许同 namespace
    - podSelector: {}        # 同 namespace 的所有 Pod
```

### 示例：允许出站但限制目标

```yaml
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: restrict-egress
spec:
  podSelector:
    matchLabels:
      app: backend
  policyTypes:
  - Egress
  egress:
  - to:
    - podSelector:
        matchLabels:
          app: database
    ports:
    - protocol: TCP
      port: 3306             # 只能访问 MySQL
  - to:
    - namespaceSelector: {}
    ports:
    - protocol: UDP
      port: 53               # 允许 DNS 查询
```
